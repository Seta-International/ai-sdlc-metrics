# 02.5 — Dynamic Tool Retrieval Inside Sub-agents

**Design §§:** §7 (Tool menu shaping — _Dynamic tool retrieval_), §3 (sub-agent declaration site — `toolRetrieval` + `coreTools`), §18.5 (12-module scale probe — EI-4/EI-5/EI-6).

**Extends:** plan 02 (sub-agent declaration site exposes `toolRetrieval` + `coreTools` config surface). This plan owns the runtime retrieval contract and its drift / observability surface.

---

## 1. Scope

### In

- Tool-descriptor embedding pipeline: embed `whenToUse + whenNotToUse + description` per agent-exposed tool at boot/load time with `text-embedding-3-small`.
- Tenant-neutral content-hash-keyed embedding store; append-only; new embedding rows on descriptor change.
- `ToolRetriever.retrieve(...)` per-invocation top-K nearest-neighbor lookup against a sub-agent's `toolScope`, unioned with the sub-agent's `coreTools` allowlist.
- Query-time embedding of the sub-agent's Phase directive (`goal + constraints`); for Tier-0 candidates, the user utterance is the query.
- Activation policy: enabled per sub-agent via `toolRetrieval.enabled`; **required** when the resolved `toolScope` contains more than 10 tools; optional below.
- Embedding-provider fallback path: on provider outage, return the full resolved `toolScope` and emit a structured signal.
- Retrieval-quality scorer on a sub-agent's golden traces; `whenToUse` token-collision authoring lint.
- Span attrs + metrics (§8) for topK resolution, fallback firing, per-sub-agent recall.

### Out

- Authoring UI or tenant-writable tool descriptors — all descriptors are code-declared via plan 01.
- Fine-tuned rerankers / Toolshed-style second-stage rerank — explicitly deferred (§17); revisited as a GA spike if the recall monitoring in §8 fires.
- Query-time re-embedding of tool descriptors — descriptors embed at load-time only; re-embeds happen on descriptor text change (new `content_hash` row).
- Semantic result cache (plan 14) — disjoint concern.
- Router-level sub-agent retrieval (owned by plan 02, R-02.26).

---

## 2. Design Context

At ~10 tools per sub-agent the accuracy cliff begins (§3, §7). The load-bearing external evidence is Anthropic's 2025-11 Tool Search release: on the internal tool-use benchmark, Tool Search lifts Claude Opus 4.5 accuracy from **79.5% → 88.1%** by surfacing only the top-K semantically relevant tools per invocation rather than inlining the entire registry. The mechanism is identical to what we need: embed each tool's natural-language descriptor once, retrieve top-K against the active directive, keep declared scope intact.

**Why not shrink `toolScope`.** Rejected. The sub-agent's capability surface is a product decision (§3 — sub-agent declaration is the invariant site). Shrinking scope to keep under the accuracy cliff amputates the sub-agent's reach; a `planner.read-write` sub-agent with a 25-tool scope is the correct declaration, not a bug to work around. Retrieval narrows _what the LLM sees per call_ without narrowing _what the sub-agent can do_.

**Why not have the router curate tools per turn.** Rejected. It entangles the router with per-domain tool semantics it shouldn't own (§3 invariant: the router produces a plan; code executes it). The router's job is sub-agent selection and directive authoring; tool selection belongs inside the sub-agent's invocation path where the directive is known.

**Why `whenToUse + whenNotToUse + description` as the embedding target.** These are the same fields the router prompt renders for sub-agents (plan 02) and that the declared-intent drift scorer consumes (plan 10, R-10.29). Reusing them keeps authoring cost linear: one descriptor serves router prompting, retrieval ranking, and drift CI.

**Why `text-embedding-3-small`.** Already in the stack (per CLAUDE.md). No new vendor dependency. Dimensionality + latency (~50ms for batch embedding at 200 tools) fits the performance budget (§10).

**Why tenant-neutral storage.** Tool descriptors are declared in code and identical across tenants; the embedding carries no tenant data. Partitioning the embedding store by tenant would duplicate rows without changing content. The open question in §18 names the alternative (tenant-partitioned for audit symmetry) for the Beta reviewer.

## 3. Data Model

### `agent_tool_embedding` (tenant-neutral, append-only)

- `tool_name TEXT` — the registered tool identifier from plan 01's `ToolRegistry`.
- `content_hash TEXT` — hash of the canonicalized `{ whenToUse, whenNotToUse, description }` descriptor.
- `embedding VECTOR` — `text-embedding-3-small` vector.
- `descriptor_snapshot JSONB` — the exact `{ whenToUse, whenNotToUse, description }` that produced the embedding (canonicalized per §8 canonicalization rules of plan 00).
- `first_seen_at TIMESTAMPTZ`.
- Primary key: `(tool_name, content_hash)`. Append-only; descriptor text changes produce a _new_ row with a new hash, never mutating the old row.
- Index: `(tool_name)` for latest-hash lookup; vector index appropriate to the driver for ANN nearest-neighbor.

Tenant-neutral rationale. Descriptors are code-declared, identical across tenants; the embedding has no tenant-specific content. Storage omits `tenant_id` and does not install RLS on this table (contrast with every other table in the codebase — this is the single documented exception, justified by the code-only, tenant-identical nature of the data). The open question in §18 names the alternative for the Beta reviewer.

### Runtime index (in-memory, derived)

- A frozen in-memory map `{ tool_name → embedding_vector }` built at `apps/api` boot from the latest-hash row per `tool_name`. Primary serving path for nearest-neighbor lookup; the DB row is the source of truth for append + replay.
- Rebuilt on deploy; never mutated at runtime.

### Descriptor load-time embedding pipeline (boot-time procedure)

Not a table; a boot-time derivation:

1. Enumerate tools from `ToolRegistry.listAgentTools()` (plan 01).
2. For each tool, canonicalize the `{ whenToUse, whenNotToUse, description }` triple + compute `content_hash`.
3. If row `(tool_name, content_hash)` absent in `agent_tool_embedding`, call `text-embedding-3-small` + append row + emit kernel audit `agent.tool_descriptor_embedded`.
4. Hydrate the in-memory index from the latest rows.

## 4. Interface Contracts

### `ToolRetriever` (module boundary consumed by sub-agent runner — plan 03)

```
retrieve(opts: {
  subAgentKey: SubAgentKey;
  directive: { goal: string; constraints: string[] };  // sub-agent Phase directive
  toolScope: ReadonlyArray<AgentToolDescriptor>;        // already-resolved, role+module+screen filtered
  coreTools: ReadonlyArray<string>;                     // from defineSubAgent config
  topK: number;                                         // from toolRetrieval.topK, default 6
}): Promise<{
  selected: ReadonlyArray<AgentToolDescriptor>;         // top-K ∪ coreTools, de-duplicated, order: coreTools first then ranked
  fallbackFired: boolean;                               // true iff embedding-provider fallback path was taken
  retrievalInputHash: string;                           // hash of the canonicalized directive query, stamped on span for replay
}>
```

- Input `toolScope` is the output of the deterministic §7 steps 1-3 menu pipeline.
- `AgentToolDescriptor` type comes from plan 01 §4.
- The union is order-sensitive: `coreTools` render first, then ranked retrieved tools; consumers relying on menu order must not re-sort.
- For Tier-0 candidate evaluation (pre-router), the caller passes the user utterance as `directive.goal` with empty constraints — same contract, different input.

### `ToolDescriptorEmbedder` (boot-time only; not a per-turn surface)

```
ensureEmbedded(descriptors: ReadonlyArray<AgentToolDescriptor>): Promise<{
  embedded: number;        // new rows appended this boot
  reused: number;          // hash-match, no call made
}>
```

### Drift artifacts

- `RetrievalQualityScorer.score(subAgentKey, goldenTraces): { recall: number; perTraceRecall: Record<TraceId, number> }` — consumed by CI (plan 10 golden-trace set).
- `WhenToUseCollisionLinter.lint(toolScope): ReadonlyArray<{ toolA: string; toolB: string; similarity: number }>` — authoring-time lint; pairs above threshold emit a warning surfaced in the PR check.

## 5. Control Flow

### Boot-time embedding (once per deploy)

1. `apps/api` boots.
2. After plan 01's `ToolRegistry` hydrates, `ToolDescriptorEmbedder.ensureEmbedded(registry.listAgentTools())` runs.
3. Canonicalize each descriptor → hash → check `agent_tool_embedding` by `(tool_name, content_hash)`. Missing rows trigger an embedding call + append + kernel audit.
4. Build the in-memory `{ tool_name → vector }` map from the latest-hash row per tool. Freeze.
5. Boot fails fast if embedding provider is unreachable AND any required embedding is missing (a degraded runtime with empty retrieval is worse than a failed deploy).

### Per-invocation retrieval (inside sub-agent runner, plan 03)

1. Sub-agent runner prepares to hand tools to the LLM for the current Phase directive.
2. If `subAgent.toolRetrieval?.enabled !== true` AND `|resolvedToolScope| ≤ 10` → bypass retrieval; surface full `resolvedToolScope`.
3. Else call `ToolRetriever.retrieve({ subAgentKey, directive, toolScope: resolvedToolScope, coreTools, topK })`.
4. Retriever canonicalizes the directive string → embeds via `text-embedding-3-small` → computes cosine similarity against the in-memory vectors for the `toolScope` subset → returns top-K.
5. Retriever unions top-K with `coreTools` (resolved from names to descriptors via registry); de-duplicates on `tool_name`.
6. Sub-agent runner renders only the union into the LLM message; records span attrs (§8).
7. LLM invokes tools as normal; the gateway (plan 01) enforces scope and permissions at call-time regardless of what the LLM sees.

### Embedding-provider outage (degraded path)

1. Retrieval call to `text-embedding-3-small` for the directive fails (timeout or non-retryable error).
2. Retriever returns `{ selected: resolvedToolScope, fallbackFired: true, retrievalInputHash }` — full scope is surfaced.
3. Metric `tool.retrieval.fallback_fired` increments; span attr `tool.retrieval.fallback: true`.
4. No user-visible degradation beyond a longer prompt; no turn failure.
5. Sustained fallback firing (alert in §8) is a platform incident, not a per-turn error.

## 7. Failure Modes & Recovery

| Failure                                                          | Symptom                                                                 | Recovery                                                                                                                               |
| ---------------------------------------------------------------- | ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| Embedding provider unreachable at boot                           | `apps/api` boot fails if any required embedding row missing             | Deploy gate catches; roll back. If provider is the issue, retry boot after provider recovers.                                          |
| Embedding provider unreachable at invocation                     | `fallbackFired: true`; full `toolScope` surfaced; metric fires          | Sustained firing alerts the platform team; per-turn behavior is degraded (longer prompt) but not failed.                               |
| Stale descriptor — tool text changed in code, not re-embedded    | Drift between active descriptor and embedded content                    | Boot-time `ensureEmbedded` always runs; a missed boot means a fresh deploy re-hydrates. Descriptor-change PRs run the drift scorer CI. |
| Retrieval recall regression on a sub-agent's golden traces       | CI hard-fail (R-02.5.8)                                                 | Author investigates: ambiguous `whenToUse`, missing negative examples in `whenNotToUse`, or a new tool with a colliding descriptor.    |
| `whenToUse` token collision between two tools                    | Lint warning on PR; retrieval ranking becomes ambiguous between the two | Author rewrites one descriptor; CI re-runs lint.                                                                                       |
| In-memory index drift between deployed tool set and embedded set | Tool exists in registry but not in index → retrieval skips it           | Boot-time `ensureEmbedded` closes the gap; a CI test asserts every registry entry has a matching embedding row before deploy.          |
| `toolRetrieval.enabled: false` with `toolScope > 10`             | Boot validator fails                                                    | Author flips flag to `true` or shrinks scope; no runtime fallback — structural invariant.                                              |

## 8. Observability Surface

### Spans

- `tool-retrieval:retrieve` — child of the sub-agent invocation span. Attrs:
  - `tool.retrieval.sub_agent_key`
  - `tool.retrieval.topk_configured`
  - `tool.retrieval.topk_resolved` — actual size of `selected` after union + de-dupe
  - `tool.retrieval.tool_scope_size` — size of `resolvedToolScope` input
  - `tool.retrieval.core_tools_size`
  - `tool.retrieval.fallback_fired: boolean`
  - `tool.retrieval.input_hash` — hash of canonicalized directive query
  - `tool.retrieval.duration_ms`

### Metrics

- `agent_tool_retrieval_invocations_total{sub_agent_key, fallback_fired}` — counter.
- `agent_tool_retrieval_duration_ms{sub_agent_key}` — histogram.
- `agent_tool_retrieval_fallback_fired_total{cause}` — counter. `cause ∈ {provider_timeout, provider_error}`. Alert at sustained > 1% of invocations per 5-minute window.
- `agent_tool_retrieval_recall_per_sub_agent{sub_agent_key}` — gauge, populated by the scheduled retrieval-quality scorer over the rolling golden-trace window.
- `agent_tool_descriptor_embedded_total{tool_name}` — counter, emitted on append.

### Dashboards

- Per-sub-agent retrieval recall (alert on sustained dip below threshold).
- Fallback-fired rate (alert on sustained breach).
- `topk_resolved` distribution — surfaces misconfigured `topK` values.

## 9. Security Considerations

- **Attack surface.** Embeddings are computed from code-declared descriptor text; no tenant data reaches the embedding provider. No cross-tenant leak surface is introduced.
- **Indirect-injection.** Tool descriptors are code-declared and reviewed in PR (plan 01 — drift test). A malicious descriptor cannot land at runtime via tenant input.
- **Directive as query.** The directive (`goal + constraints`) is authored by the router LLM and may contain tenant-influenced text. It is sent to the embedding provider as a retrieval query but is not stored by the provider's default configuration; verify provider settings per stack-default. Directive content is also present in the router plan (already embedded via §3 invariants), so this does not expand the exfiltration surface.
- **Tenant-neutral store exception.** Explicitly justified (§3): descriptors are code-only and identical across tenants. The exception is documented in code-review checklists so no other table sneaks into the "tenant-neutral" pattern without challenge.

## 10. Performance Budget

| Operation                                              | p50    | p95    | p99                    |
| ------------------------------------------------------ | ------ | ------ | ---------------------- |
| In-memory nearest-neighbor over ≤200-tool surface      | <5ms   | <10ms  | <15ms                  |
| Directive embedding (`text-embedding-3-small`, single) | <20ms  | <35ms  | <50ms (provider-bound) |
| `ToolRetriever.retrieve` end-to-end                    | <25ms  | <45ms  | **<50ms** target       |
| Boot-time `ensureEmbedded` (cold, 200 new tools)       | —      | —      | <5s (batch embed)      |
| Boot-time `ensureEmbedded` (warm, zero new)            | <100ms | <200ms | <500ms (DB reads)      |

Budget rationale: the p99 retrieval target <50ms keeps retrieval negligible against a router-LLM call (p95 ~5s, plan 02). If the directive embedding provider degrades, the fallback path (§7 R-02.5.7) absorbs the latency spike by returning full scope immediately.

## 11. Testing Strategy

### Unit

- `ToolRetriever.retrieve` returns exactly `topK` ranked tools (pre-union) for a seeded directive + vector fixture.
- Union semantics: `coreTools` always present regardless of similarity ranking; de-duplicated on `tool_name`; order `coreTools` first then ranked.
- Fallback path: inject embedding-provider error → `fallbackFired: true`, `selected === resolvedToolScope`.
- Boot-time validator: seed a sub-agent with `toolScope > 10` AND `toolRetrieval.enabled: false` → boot fails.
- `WhenToUseCollisionLinter` flags a seeded near-duplicate pair; inverse fixture with distinct descriptors returns empty.

### Integration

- Boot a real `apps/api` with a fresh embedding store → `ensureEmbedded` populates all rows, kernel audit events present, in-memory index non-empty.
- Re-boot the same build → zero new embedding calls; `reused` count equals tool count.
- Descriptor text change → new `(tool_name, content_hash)` row appended; old row preserved; in-memory index uses new hash.
- Embedding provider outage at invocation → sub-agent turn completes; fallback metric increments; span attr present.
- 12-sub-agent synthetic probe (EI-5): seed the fixture registry, run the retrieval-quality scorer, assert recall ≥ target.

### Property

- Union de-duplication: for any random `coreTools ⊂ toolScope` and any random `topK`, result size ≤ `|coreTools| + topK` and every `coreTool` present.
- Determinism: same directive + same vector index → same top-K ranking.
- Order invariant: `selected[0..|coreTools|-1]` is always the resolved `coreTools` in declaration order.

### Fixtures

- `fixtures/retrieval/12-sub-agent-probe/` — EI-5 synthetic fixture with ≥20 tools per sub-agent and a labeled golden-trace set.
- `fixtures/retrieval/collision-pair.ts` — two tools with near-duplicate `whenToUse`; lint expected to flag.
- `fixtures/retrieval/provider-outage.ts` — embedding call throws; fallback path expected.

## 12. Acceptance Criteria

- All unit + integration + property tests pass.
- 12-sub-agent synthetic probe (EI-5): retrieval recall ≥ target; CI hard-fails on regression.
- Seeded `whenToUse` collision pair is flagged by the lint in a dry-run PR.
- Boot-time validator hard-fails a seeded sub-agent with `toolScope > 10` AND `toolRetrieval.enabled: false`.
- Exported trace for a sub-agent invocation shows `tool-retrieval:retrieve` child span with `topk_resolved`, `tool_scope_size`, `fallback_fired`, and `input_hash` attrs populated.
- Embedding-provider outage injection: integration test confirms `fallbackFired: true`, turn completes, metric increments.
- Descriptor text change PR produces a new `(tool_name, content_hash)` row in `agent_tool_embedding`; old row untouched.

## 13. Rollout Plan

- **Phase 1** — ship `ToolDescriptorEmbedder` + `agent_tool_embedding` table; run embedding pipeline at boot against the MVP tool surface. Retriever present but not yet consumed (sub-agents below 10-tool threshold — structurally dormant).
- **Phase 2** — ship `ToolRetriever` wired into the sub-agent runner (plan 03). Activation remains structural: only sub-agents whose `toolScope > 10` invoke retrieval; the MVP 3-module case leaves retrieval dormant.
- **Phase 3** — first sub-agent declaration exceeds the 10-tool threshold; retrieval activates naturally; monitoring dashboards populate.
- **Phase 4** — enable retrieval-quality scorer as a CI hard-gate once the golden-trace set for the relevant sub-agent is populated.

**Backout.** Retrieval can be disabled per sub-agent by flipping `toolRetrieval.enabled: false` — but only if `toolScope ≤ 10` (the validator enforces this). For a retriever bug affecting a sub-agent whose scope exceeds 10, the back-out is to shrink `toolScope` in the same PR or roll the PR back. No runtime feature-flag: activation is structural, not flag-driven (CLAUDE.md "no shims" rule).

## 16. Activation Gate

MVP — ships with first production turn. Retrieval is **structurally dormant** below the 10-tool threshold: the MVP 3-module sub-agent set does not exceed it, so the retriever is wired but not invoked. Activation happens automatically when the first sub-agent's `toolScope` crosses 10 tools — no flag, no config change, no rollout ceremony. The boot-time validator enforces the correctness of the activation decision.

The 12-sub-agent synthetic probe (EI-5) exercises the active path continuously in CI so the code is hot-tested before production traffic reaches the threshold.

## 17. Out of Scope

- **Fine-tuned rerankers / Toolshed-style second-stage rerank.** The Anthropic Tool Search baseline (79.5% → 88.1%) is the MVP target; richer rerank strategies are a GA spike if the retrieval-recall monitoring (§8) fires a sustained regression signal. Revisited per §16 of the runtime spec.
- **Query-time re-embedding of tool descriptors.** Descriptors embed at load time only. Re-embeds happen on descriptor text change (new `content_hash` row), not per-turn. Query-time re-embedding of _the directive_ is in-scope and happens every invocation.
- **Router-level sub-agent retrieval.** Lives in plan 02, R-02.26.
- **Semantic result cache.** Disjoint concern; owned by plan 14.
- **Tenant-writable descriptors.** Descriptors are code-declared. Admin UI for descriptor overrides is not on the roadmap; if ever introduced, it requires a fresh security review against the indirect-injection surface (§9).

## 18. Open Questions

- **Tenant-neutrality of the embedding store.** Default is tenant-neutral (no `tenant_id`, no RLS) because descriptors are code-only and identical across tenants. The alternative — tenant-partition the store for audit symmetry with every other table in the codebase — is cheaper on reviewer cognition ("every table has `tenant_id`, no exceptions" per CLAUDE.md) but duplicates rows without changing content. Owner: security reviewer at Beta entry; default survives unless audit policy demands uniformity.
- **`topK` default of 6.** Seeded from the §7 design default. Re-tune after the first sub-agent crosses the 10-tool threshold and produces production retrieval traces. Owner: runtime lead at first-activation review.
- **Retrieval-recall target threshold.** The 12-sub-agent synthetic probe needs a concrete numeric target for CI. Seed from Tool Search's 88.1% accuracy as an upper-bound reference point; the real target depends on the labeled-golden-trace set size and is finalized with plan 10. Owner: harness maintainer.
- **`whenToUse` collision similarity threshold.** The lint's pass/fail boundary needs empirical tuning against seeded collisions in the 12-sub-agent probe. Owner: authoring-governance reviewer at Beta entry.
- **Provider-side retention of directive queries.** The embedding call ships the directive to `text-embedding-3-small`. Provider default retention policy must be verified (provider-side logs = tenant-influenced content exposure). Owner: security reviewer before first activation.
