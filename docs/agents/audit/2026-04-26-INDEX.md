# Agent Runtime Production-Readiness Audit — Master Index

**Date:** 2026-04-26
**Scope:** Plans 00–15 (Plan 16 / code-execution composition tier deliberately excluded — out of scope for AaaS).
**Method:** Read-only verification audit by 5 parallel Sonnet subagents, one per cluster, full §1–§18 walk per plan.
**Severity scale:** P0 (production-blocking) · P1 (drift) · P2 (quality) · INFO (intentional deferral per §17/§18 or README beta-gate).

> **Headline:** The agent runtime has well-architected scaffolding (DDD boundaries, ports/adapters, OTel surface design, draft tier classification, governance lints) but the **intelligence layer, the turn-startup wiring, the async execution path, and the CI quality gates are not connected to production code paths.** Several plans marked _Shipped_ in the README are code-complete but functionally inert. The CLAUDE.md hard rule **"Every table has tenant_id"** is honoured at the schema level but not at the RLS-enforcement level — only **1 of ~30** agents tables actually has `ENABLE / FORCE ROW LEVEL SECURITY`.

## 2 — Cross-Cluster Themes (Read These First)

These are the issues that surface in **multiple clusters** and cannot be fixed by patching one plan in isolation. Together they explain why nominally-shipped plans don't function in production.

### Theme A — Systemic RLS gap (10 P0 findings across C1, C2, C4, C5)

**Symptom:** Only `agents.agent_tool_result_cache` has `ENABLE / FORCE ROW LEVEL SECURITY` and a tenant-isolation policy in `0000_initial.sql` (lines 1832–1836). Every other tenant-scoped table — ~28 of them, including `agent_conversation`, `agent_message`, `agent_l3_preference`, `agent_scratchpad`, `agent_draft`, `agent_schedule`, `agent_schedule_run`, `core.agent_delegation`, `agent_canary_run`, `agent_canary_query`, `agent_golden_trace`, `agent_rollout_config`, `agent_rollout_event`, `agent_shadow_run`, `agent_tenant_budget`, `agent_user_budget`, `agent_rate_limit_counter`, `agent_active_turn`, `agent_turn_sampling_decision`, `agent_cost_event`, `agent_insight`, `agent_chat_session`, `agent_chat_message`, `agent_iteration`, `agent_session`, `agent_stored_sub_agent`, `agent_tool_invocation`, `agent_prompt_store`, `agent_narrative_store` — has **no** RLS DDL.

**Why this happened:** Drizzle's `pgSchema.table()` does not emit `ENABLE ROW LEVEL SECURITY` — it must be added as raw SQL in the migration. Plan 14 added it for its single new table; no other plan did.

**Why it matters now:** Plan 04 R-04.4 / R-04.15, Plan 08 §9, Plan 10 §9, Plan 11 §9 all _claim_ tenant isolation is enforced "by RLS". Integration tests assert `relforcerowsecurity=true` (e.g. `drizzle-conversation.repository.integration.spec.ts:360`) — they will fail in CI the moment they actually run. The RlsMiddleware (which sets `app.tenant_id` per request) is wired but does nothing without the policies.

**One fix unblocks at least 10 P0s.**

### Theme B — Turn-controller wiring black hole (3 P0 in C3)

`apps/api/src/modules/agents/interface/http/agent-turn-controller.ts:87` is the entry point for every live turn, and it currently calls **none of**:

- `BudgetChecker.preTurnCheck()` — Plan 05 R-05.1 (budget enforcement)
- `ObservabilityContextFactory.create()` — Plan 07 R-07.43 (root span / flow_id)
- `FlowIdPropagation.mint()` — Plan 07 R-07.44 (flow correlation)

The classes exist, are tested, and have clean interfaces. They just aren't injected into the controller. Result: every turn runs uncapped, untraced, and unmetered. Plans 05 and 07 are marked **Shipped** but produce zero spans and zero budget enforcement in live traffic.

**Estimated effort:** ~1 day; closes 3 P0s and ~6 P1s.

### Theme C — Core intelligence layer is stubbed (3 P1 in C1, but the most consequential drift)

- `sub-agent-runner-adapter.ts:72` — returns `rawStructured: {}` with all-zero signals. The Vercel AI SDK ReAct loop is **not wired**. Sub-agents never call any tool.
- `synthesizer-adapter.ts:23` — concatenates sub-agent summaries with `join(' ')`. No LLM call. Final answers are mechanical concatenation.
- `sub-agent-runner.ts:160` — `usageTotals` hardcoded to `ZERO_USAGE` in the success branch. All cost / token telemetry is identically zero.

These are the **only thing the system is supposed to do**: run sub-agents, call tools, synthesize. The infrastructure around them is production-grade; the work itself is not happening.

(Note: cluster 1's audit assigned these P1 because the plan declared the seam shape was correct. By any operational definition these are P0 — adjusted in the prioritized list below.)

### Theme D — Async / scheduled path is dry-run only (1 P0 in C4)

`scheduled-turn-worker.ts:100–126` — `handle()` emits a dry-run audit and marks the run `'completed'` without invoking the turn pipeline. Plan 09's MVP scope (read-only + draft-to-inbox) requires the worker to actually run a read-only turn. Currently all scheduled turns produce an audit log entry and nothing else.

### Theme E — CI quality gates are theatrical (3 P0 in C5)

- `golden-trace-runner.ts:142` — `runCiGate` sets `actualFingerprint = expectedFingerprint`. The CI gate that's supposed to catch regressions in all 15 other plans always passes.
- `extensibility-invariant-audit.ts:206` — EI-7, EI-8, EI-9, EI-10 return `passed:true` unconditionally. **4 of 10** extensibility invariants are blind.
- `leak-canary.scheduler.ts` — daily cross-tenant leak canary always records `'clean'` without scanning. Provides false security assurance.

The PR review CI gate, the EI audit, and the leak canary are wired structurally but produce no real signal. The README implies they are operational.

### Theme F — Shadow mode is unverifiable (2 P0 in C5)

- `trpc-caller.ts:60` — `mode:'dry-run'` throws `Error("not supported at MVP")`. Plan 11's entire shadow-execution premise depends on dry-run.
- `shadow-turn-worker.ts:108` — `simulateShadowExecution()` returns `null`. The §12 acceptance criterion — "seeded adversarial test proves no real writes commit under mode:dry-run" — cannot be met because dry-run doesn't run.
- `shadow-turn-worker.spec.ts` — the seeded adversarial test is also missing. There is no proof, structural or empirical, that shadow turns are side-effect-free.

### Theme G — Kernel audit events incomplete (1 P0 in C4)

Plan 08 R-08.28 names **seven** kernel audit events. Three (`agent.draft_approved`, `agent.draft_rejected`, `agent.draft_execution_failed`) are entirely absent — zero grep hits across the agents module. Approval / rejection / execution-failure paths emit nothing.

### Theme H — GDPR pipeline incomplete (1 P0 in C2)

- `gdpr-erasure.ts:111` — `l35ScratchpadDeleted` hardcoded to `1`. Compliance audit log fabricates a count regardless of actual deletions.

(The Langfuse `purgeByUserId` step previously listed here has been removed by user directive: Langfuse is not deployed at MVP. Trace backend purge-by-user semantics are deferred to when a trace backend is selected.)

### Theme I — OTel metric instruments not defined (3 P0 in C3)

Plan 05 §8 names 9 metric instruments (`agent_cost_usd_total`, `agent_tier_shift_total`, `agent_provider_fallback_total`, `agent_budget_remaining_usd`, `agent_llm_call_attempt_duration_ms`, `agent_rate_limit_rejection_total`, `agent_approval_inbox_throttle_total`, `agent_budget_ceiling_hit_total`, `agent_degradation_step_total`). Plan 06 §8 names 8 more. **None of the 17 instruments exist.** `gateway-metrics.ts` covers Plans 01/02/12/14 only.

### Theme J — DB schema drift (3 P1 in C1, C2, C4 plus 1 P0 in C2)

- `agent_tool_embedding` uses `JSONB` not `vector`; no pgvector ANN index → tool retrieval is O(n) JS linear scan.
- `agent_message` has no GIN FTS index despite repository performing inline `to_tsvector` queries → full-table scan FTS in production.
- `agent_tool_result_cache` has no unique constraint on `(tenant_id, tool_name, canonical_args_hash)`; `onConflictDoNothing()` has no target → concurrent puts insert duplicates → cache exact lookup may return multiple rows.
- Kernel `audit_event` has no `flow_id` / `intent_slug` columns, blocking Plan 07 §3 prerequisite.

## 5 — Cross-Cluster Seam Observations

These are integration concerns that no single cluster owns:

- **Plan 04 L3.5 scratchpad ↔ Plan 12 iterative loop.** Plan 12's iterative supervisor is supposed to read/write L3.5 across iterations. Cluster 1 found Plan 12 wiring is partial; cluster 2 found L3.5 schema is correct but `Scratchpad.deleteForUser` returns void (not a count). The seam is intact in shape but neither side stress-tests the other.
- **Plan 07 flow_id ↔ all kernel audit events.** Plan 07 requires every kernel audit event to carry `flow_id` and `intent_slug`. The kernel `audit_event` schema doesn't have those columns yet. Every plan emitting kernel audit events (08, 09, 13, 14, 15) will need to update its `KernelAuditFacade.recordEvent()` calls once the kernel migration lands.
- **Plan 14 semantic cache ↔ Plan 04 L3 preferences.** Plan 14's per-tenant semantic cache uses the same pgvector pattern that Plan 04's `agent_semantic_index_<tenant>` would use. Plan 14 has the unique-constraint gap (Theme J); Plan 04 has the schema-not-yet-created gap. Implementing Drizzle pgvector once for Plan 14 unblocks Plan 04's index.
- **Plan 08 drafts ↔ Plan 11 shadow.** Plan 08's draft execution path and Plan 11's shadow execution path both depend on a non-side-effect mode in tRPC. Plan 11 found `mode:'dry-run'` throws; Plan 08 found execute-approved-draft skips revalidation. A unified mode-aware tRPC caller closes both.
