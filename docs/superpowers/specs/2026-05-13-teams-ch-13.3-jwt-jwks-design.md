# WBS 13.3 — JWT/JWKS Verifier for Inbound Activities

**Date:** 2026-05-13
**Updated:** 2026-05-14
**Status:** Draft
**Author:** AG-F1 (with Claude)

---

## Context: what changed after 13.1 merged

- Package is now `@seta/ms-teams` (was `@seta/teams`).
- `platform/auth` (`@seta/auth`) exists with `auth.users`, `auth.sessions`, `auth.api_keys` schema — but **no `jwks_cache` table yet**. The table is added here.
- After 13.2, the inline `app.post('/teams/messages', ...)` in `main.ts` is gone — `routes.ts` now owns the endpoint. Removing `skipJwtVerify` in this task means updating `routes.ts` and `apps/api/src/env.ts` only.

---

## Goal

Every inbound `POST /teams/messages` is JWT-authenticated before the activity is parsed. Remove `TEAMS_SKIP_JWT_VERIFY` entirely. After this task, the channel behaves identically in dev and prod — both require a valid Bot Framework JWT.

---

## Background: why three patterns are required

From `docs/setup.md` §7 (pinned `jose` 6.x patterns — do not deviate):

1. `createRemoteJWKSet` takes a **URL object**, not a string. Using a string silently breaks in jose 6.x.
2. `jwtVerify` must pin `algorithms: ['RS256']` to block "none"/HS-confusion attacks.
3. During MS key rotation, two valid keys briefly match the same `kid` — `ERR_JWKS_MULTIPLE_MATCHING_KEYS` must be caught and retried, otherwise valid tokens are rejected for several minutes.

### New: `modules/channels/teams/src/jwt.ts`

```ts
import * as jose from 'jose'
import { eq } from 'drizzle-orm'
import { db } from '@seta/db'
import { jwksCache as jwksCacheTable } from '@seta/auth'
import { BotFrameworkJwtInvalid } from './errors.js'
import { logger } from '@seta/observability'

const BOT_FRAMEWORK_JWKS_URL = new URL('https://login.botframework.com/v1/.well-known/keys')
const BOT_FRAMEWORK_ISSUER   = 'https://api.botframework.com'
const CACHE_KEY               = 'botframework'

// Module-level singletons — initialised once per process lifetime
let JWKS: ReturnType<typeof jose.createRemoteJWKSet> | undefined
let jwksCacheState: jose.JWKSCacheInput = {}

async function initJwks(): Promise<ReturnType<typeof jose.createRemoteJWKSet>> {
  if (JWKS) return JWKS
  jwksCacheState = await readJwksCacheFromDb() ?? {}
  JWKS = jose.createRemoteJWKSet(BOT_FRAMEWORK_JWKS_URL, {
    cooldownDuration: 30_000,              // throttle refetches on unknown kid
    [jose.jwksCache]: jwksCacheState,
  })
  return JWKS
}

export async function verifyBotFrameworkJwt(
  token: string,
  botId: string,
): Promise<jose.JWTPayload> {
  const jwks = await initJwks()
  const prevUat = jwksCacheState.uat
  const verifyOpts: jose.JWTVerifyOptions = {
    issuer:         BOT_FRAMEWORK_ISSUER,
    audience:       botId,
    algorithms:     ['RS256'],             // pin algs — never allow 'none' or HS variants
    clockTolerance: 60,                    // Bot Framework clocks drift up to ~60 s
  }

  try {
    const { payload } = await jose.jwtVerify(token, jwks, verifyOpts)
    if (jwksCacheState.uat !== prevUat) await writeJwksCacheToDb(jwksCacheState)
    return payload
  } catch (err) {
    if (
      err instanceof Error &&
      'code' in err &&
      (err as { code: string }).code === 'ERR_JWKS_MULTIPLE_MATCHING_KEYS'
    ) {
      // MS key rotation: two valid keys briefly share the same kid.
      // jose tries the next matching key on a second invocation.
      logger.warn('JWKS multi-match during key rotation — retrying')
      const { payload } = await jose.jwtVerify(token, jwks, verifyOpts)
      if (jwksCacheState.uat !== prevUat) await writeJwksCacheToDb(jwksCacheState)
      return payload
    }
    throw new BotFrameworkJwtInvalid(err instanceof Error ? err.message : String(err))
  }
}

async function readJwksCacheFromDb(): Promise<jose.JWKSCacheInput | null> {
  const row = await db.query.jwksCache.findFirst({
    where: eq(jwksCacheTable.key, CACHE_KEY),
  })
  return row ? (row.payload as jose.JWKSCacheInput) : null
}

async function writeJwksCacheToDb(cache: jose.JWKSCacheInput): Promise<void> {
  await db
    .insert(jwksCacheTable)
    .values({ key: CACHE_KEY, payload: cache, updatedAt: new Date() })
    .onConflictDoUpdate({
      target: jwksCacheTable.key,
      set: { payload: cache, updatedAt: new Date() },
    })
}
```

## Tests (`modules/channels/teams/src/jwt.test.ts`)

All external HTTP via msw recordings in `__recordings__/jwt/`. No live MS endpoints in CI.

| Test | Setup | Expected |
|---|---|---|
| Valid JWT | msw serves correct JWKS; token signed with matching key | returns payload |
| Expired token | token `exp` in the past | throws `BotFrameworkJwtInvalid` |
| Wrong `aud` | token `aud` != botId | throws `BotFrameworkJwtInvalid` |
| Wrong `iss` | token `iss` != Bot Framework issuer | throws `BotFrameworkJwtInvalid` |
| Unknown `kid` | msw: first JWKS response lacks kid; second includes it | retries JWKS fetch; returns payload |
| Multi-matching `kid` | msw: raises ERR_JWKS_MULTIPLE_MATCHING_KEYS then resolves | retry succeeds; returns payload |
| JWKS cache write-back | `cache.uat` changes after verify | `writeJwksCacheToDb` called (spy) |
| Cache read on init | DB row exists | `createRemoteJWKSet` receives pre-populated cache |

Integration test (`tests/integration/jwt-cache.test.ts` — requires `DATABASE_URL`):
- Verify → assert `auth.jwks_cache` row written.
- Re-init module → assert JWKS cache read from DB (no remote fetch on warm start).
