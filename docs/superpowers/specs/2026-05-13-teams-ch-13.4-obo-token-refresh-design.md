# WBS 13.4 — OBO Token Refresh on User-Context Calls

**Date:** 2026-05-13
**Status:** Draft
**Author:** FS (with Claude)

---

## Goal

When Teams completes SSO sign-in for a user, exchange the SSO token for a Graph OBO token and store it in `oauth.oauth_tokens`. The mock handler continues returning hardcoded cards — OBO just ensures the token vault is populated so 13.5 (the real Planner handler) can retrieve a valid Graph token without any additional auth ceremony.

---

## How Teams SSO triggers OBO

Teams sends a `signin/tokenExchange` invoke activity automatically on the user's first interaction with the bot, provided the manifest includes `webApplicationInfo`. The sequence:

```
1. User sends a message in Teams 1:1 chat with SetaAgent
2. Teams intercepts — user not yet authenticated
3. Teams sends invoke { name: 'signin/tokenExchange', value: { token: <AAD SSO token> } }
4. sso.ts calls provider.acquireOnBehalfOf({ tenantId, userAssertion: ssoToken, scopes }) → vault.put()
5. MSAL exchanges SSO token → OBO access token + refresh token
6. Tokens upserted into oauth.oauth_tokens (single-flight, SELECT … FOR UPDATE)
7. Bot returns InvokeResponse { status: 200 }  ← success signal to Teams
8. Teams re-sends the original user message
9. Mock handler responds with a scripted card as before
```

If step 4 fails (admin consent not yet granted), the bot returns `{ status: 412 }` and Teams shows a "Sign in" button.

## Files

### New: `modules/channels/teams/src/sso.ts`

```ts
import type { OAuthProvider, TokenVault } from '@seta/oauth'
import { logger } from '@seta/observability'
import type { Activity, OutboundActivity } from './index.js'

const PLANNER_SCOPES = [
  'https://graph.microsoft.com/Tasks.ReadWrite',
  'https://graph.microsoft.com/User.Read',
]

export async function handleSsoTokenExchange(
  activity: Activity,
  tenantId: string,
  deps: { provider: OAuthProvider; vault: TokenVault },
): Promise<OutboundActivity> {
  const value = activity.value as Record<string, unknown> | undefined
  const userAssertion = typeof value?.token === 'string' ? value.token : undefined

  if (!userAssertion) {
    logger.warn({ conversationId: activity.conversation.id }, 'signin/tokenExchange missing token')
    return { type: 'invokeResponse', value: { status: 400 } }
  }

  try {
    // Exchange Teams SSO token for Graph OBO token via MSAL
    const bundle = await deps.provider.acquireOnBehalfOf({
      tenantId,
      userAssertion,
      scopes: PLANNER_SCOPES,
    })

    // Store in vault so future Graph calls (13.5+) can retrieve it without re-auth
    const homeAccountId = bundle.meta.homeAccountId as string
    await deps.vault.put(tenantId, 'entra', `user:${homeAccountId}`, bundle)

    return { type: 'invokeResponse', value: { status: 200 } }
  } catch (err) {
    // Consent not yet granted — Teams will show the "Sign in" prompt
    logger.warn({ err, tenantId }, 'OBO exchange failed — likely missing admin consent')
    return { type: 'invokeResponse', value: { status: 412 } }
  }
}
```

## `OAuthProvider.acquireOnBehalfOf` contract

Defined in `platform/oauth/src/provider.ts` (Epic 1). Actual signature:

```ts
acquireOnBehalfOf(input: {
  tenantId:      string    // seta tenant id (uuid)
  userAssertion: string    // the Teams SSO token from signin/tokenExchange
  scopes:        string[]
}): Promise<TokenBundle>
// Returns { accessToken, refreshToken, meta: { homeAccountId, ... } }
// Does NOT store — caller is responsible for vault.put()
```

The storage pattern used in `sso.ts` (mirrors `platform/oauth/src/routes.ts` exchange-obo endpoint):

```ts
const bundle = await deps.provider.acquireOnBehalfOf({ tenantId, userAssertion, scopes })
await deps.vault.put(tenantId, 'entra', `user:${bundle.meta.homeAccountId as string}`, bundle)
```

## Single-flight guarantee (for reference)

`vault.put` uses the MSAL `ConfidentialClientApplication` token cache (per-tenant LRU in `platform/oauth`). If Teams retries `signin/tokenExchange` before the first call completes, MSAL's internal locking prevents a duplicate MSAL round-trip. The Teams channel does not need additional locking beyond what the vault and MSAL provide.

## Tests (`modules/channels/teams/src/sso.test.ts`)

All external HTTP via msw recordings in `__recordings__/sso/`. `@seta/oauth.acquireOnBehalfOf` is called through its real export — **not mocked** (per CLAUDE.md: never mock internal `@seta/*` modules).

| Test | Setup | Expected |
|---|---|---|
| Valid SSO token, consent granted | msw: MSAL OBO endpoint returns access_token | `acquireOnBehalfOf` completes; returns `{ status: 200 }` |
| OBO fails — no consent | msw: MSAL returns `invalid_grant` | returns `{ status: 412 }`; no token stored |
| Missing `token` in value | activity.value has no `token` field | returns `{ status: 400 }` |
| Non-SSO invoke routed to handler | activity.name = 'adaptiveCard/action' | `handler(activity, runCtx)` called; SSO handler not called |

- Full `signin/tokenExchange` activity → `oauth.oauth_tokens` row written.
- Replay of same activity (idempotent) → row updated, not duplicated.
