# Plan 0 — `@seta/agent-vector` citation-span column (PR 1)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a nullable `span jsonb` column to `agent_vector.chunks`. Extend `searchChunks` to `SELECT source_id, span` and return them on `SearchHit`. Thread `span` through `insertChunks` with a small Zod refinement at the boundary. Update integration tests. Companion to the `@seta/agent-rag` package (EP-07); must merge before Plan A starts.

**Architecture:** Single Postgres schema change (`ALTER TABLE … ADD COLUMN span jsonb`). Drizzle schema edits drive `drizzle-kit generate` to emit a forward-only migration. Two API surfaces touched: `SearchHit` gains `sourceId` + `span`; `NewChunk` gains `span`. No HNSW / RLS / dedup changes.

**Tech Stack:** TypeScript (ESM), Drizzle ORM 0.45.2 + drizzle-kit 0.31.10, postgres-js 3.4.9, pgvector 0.8.2, Zod 4.4.3, Vitest, pnpm 11.

**Spec:** [`docs/superpowers/specs/2026-05-18-agent-vector-span-citation-design.md`](../../specs/2026-05-18-agent-vector-span-citation-design.md)

- [ ] **Step 3: Add the `span` column to the table definition**

Find the existing `tokenCount` line inside the `chunks` table columns object:

```ts
tokenCount: integer('token_count').notNull(),
embedding: vector('embedding', { dimensions: EMBEDDING_DIMENSIONS }).notNull(),
```

Insert the `span` column between `tokenCount` and `embedding`:

```ts
tokenCount: integer('token_count').notNull(),
span: jsonb('span').$type<{ startChar: number; endChar: number } | null>(),
embedding: vector('embedding', { dimensions: EMBEDDING_DIMENSIONS }).notNull(),
```

Notes:
- No `.notNull()` — column is nullable by design (legacy rows keep `NULL`).
- `$type<…>()` narrows the inferred TS type. `$inferSelect.span` becomes `{ startChar: number; endChar: number } | null` (the union with `null` is added automatically because the column is nullable).
- Column order matters only for the generated SQL; functionally irrelevant.

## Task 4: Thread `span` through `insertChunks` with Zod refinement

**Files:**
- Modify: `platform/agent/vector/src/ingest.ts`

- [ ] **Step 1: Read the current file**

```powershell
Get-Content platform/agent/vector/src/ingest.ts
```

Note the `for (const r of rows)` validation loop (currently checks `tenantId` only) and the `values` mapper. We add Zod refinement to the loop and a `span` field to the values mapper.

- [ ] **Step 2: Import Zod and define the span schema**

At the top of the file, after the existing imports, add:

```ts
import { z } from 'zod'

const spanSchema = z
  .object({
    startChar: z.number().int().min(0),
    endChar: z.number().int().min(1),
  })
  .refine((s) => s.endChar > s.startChar, {
    message: 'endChar must exceed startChar',
  })
  .nullable()
```

`endChar` is exclusive (matches `@seta/agent-chunking`'s `Chunk` convention — half-open `[startChar, endChar)`). `startChar` may be 0; `endChar` must be at least 1 *and* strictly greater than `startChar`.

- [ ] **Step 3: Add the Zod check inside the validation loop**

Find:

```ts
  for (const r of rows) {
    if (r.tenantId !== tenantId) {
      throw new VectorInsertFailedError(
        new Error(`row tenantId ${r.tenantId} does not match context tenantId ${tenantId}`),
      )
    }
  }
```

Replace with:

```ts
  for (const r of rows) {
    if (r.tenantId !== tenantId) {
      throw new VectorInsertFailedError(
        new Error(`row tenantId ${r.tenantId} does not match context tenantId ${tenantId}`),
      )
    }
    spanSchema.parse(r.span ?? null)
  }
```

`spanSchema.parse` throws `ZodError` on bad shape. Per the spec we deliberately do **not** wrap it in `VectorInsertFailedError` — Zod failures here indicate a programmer bug at the boundary, not a database failure, and the upstream caller (`@seta/agent-rag`) already supplies valid spans.

- [ ] **Step 4: Pass `span` through to the SQL values**

Find:

```ts
      const values = rows.map((r) => ({
        tenant_id: r.tenantId,
        source_id: r.sourceId,
        content: r.content,
        content_hash: r.contentHash,
        token_count: r.tokenCount,
        embedding: vectorLiteral(r.embedding as number[]),
      }))
```

Replace with:

```ts
      const values = rows.map((r) => ({
        tenant_id: r.tenantId,
        source_id: r.sourceId,
        content: r.content,
        content_hash: r.contentHash,
        token_count: r.tokenCount,
        span: r.span ?? null,
        embedding: vectorLiteral(r.embedding as number[]),
      }))
```

`postgres@3.4.9` auto-serializes a JS object to `jsonb` when the target column type is `jsonb` — no explicit `JSON.stringify`.
