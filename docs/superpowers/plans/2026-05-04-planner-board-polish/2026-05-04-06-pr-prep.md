# Planner Board Polish — Plan 06: Test Coverage & PR Prep

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Verify ≥70% test coverage across the changed files, fix any gaps, confirm the full build is green, run a manual checklist of the golden paths, and create the pull request.

**Architecture:** This plan has no new feature code. It wires together the five preceding plans (01–05) with a final quality gate: coverage threshold, typecheck, unit suite, and a structured manual testing checklist before the PR lands.

**Tech Stack:** Vitest coverage (v8), TypeScript, `gh` CLI for PR creation

**Spec source:** `docs/superpowers/specs/2026-05-04-planner-board-polish-design.md` — all sections; CLAUDE.md §Testing

### Task 3: Manual QA checklist

Run against `http://localhost:3011/plans/<any-plan>/board` with the dev server running. Tick each item before proceeding to PR creation.

- [ ] **Step 8: Start the dev stack**

  ```bash
  # In a separate terminal:

  ```

  Navigate to a plan board in the browser.

- [ ] **Step 9: Execute the golden paths**

  Check each item manually:

  **Progress icon (Plan 02)**
  - [ ] Task with `progress=0` shows a dashed grey circle
  - [ ] Task with `progress=50` shows an amber half-circle
  - [ ] Task with `progress=100` shows a green circle with a dark checkmark

  **Priority icon (Plan 02)**
  - [ ] Priority Normal shows a horizontal grey dash
  - [ ] Priority Low shows 2 filled grey bars + 1 dim bar (3 total)
  - [ ] Priority Important shows 3 bright bars (all filled)
  - [ ] Priority Urgent shows an amber square with a `!`

  **Mutation bugs (Plan 03)**
  - [ ] Set priority on a task card → close picker → set priority again → no 409 stale version error in console
  - [ ] Set due date by typing a full date → tab out → date is saved; partially typed date does not fire mutation
  - [ ] Apply a label → remove the label → apply again → no 409 error
  - [ ] Progress toggle is always visible (no hover-only opacity class)

  **Column UI (Plan 04)**
  - [ ] Add Task button has a dashed border and semi-transparent background (not a ghost button)
  - [ ] Empty bucket shows the kanban SVG, "Nothing to review" text, and a description
  - [ ] Column `+` shortcut button appears in the column header and opens QuickAddTask

  **Drag-drop (Plan 05)**
  - [ ] Drag a task card to a different column → task moves immediately (optimistic) → reload → task is in new column
  - [ ] Drag a column → columns reorder → reload → order persists
  - [ ] Enable sort (e.g., append `?sort=title:asc` to URL) → sort-active chip appears → dragging a card within the same column does nothing → dragging to a different column still works

### Task 4: Create the pull request

- [ ] **Step 10: Push the branch**

  ```bash
  git push -u origin HEAD
  ```

- [ ] **Step 11: Create the PR**

  ```bash
  gh pr create \
    --title "feat(web-planner): planner board polish — icons, mutation fixes, drag-drop, column UI" \
    --body "$(cat <<'EOF'
  ## Summary

  Implements the [Planner Board Polish design spec](docs/superpowers/specs/2026-05-04-planner-board-polish-design.md) across six plans:

  - **Plan 01** — API handlers now return `{ updatedAt: Date }` for `setPriority`, `setDates`, `applyLabel`, `removeLabel`; fixes stale `expectedVersion` 409 on rapid mutations
  - **Plan 02** — `ProgressIcon` (dashed/amber/dark-checkmark) and `PriorityIcon` (4 distinct SVG shapes) redesigned to match spec; hardcoded colors, no Tailwind color token classes
  - **Plan 03** — TaskCard and LabelPicker write server `updatedAt` to cache after each mutation; due date uses `onBlur` instead of `onChange` to prevent mid-input fires; progress toggle always visible
  - **Plan 04** — `QuickAddTask` Add Task button has dashed border + semi-transparent background; empty bucket state with kanban SVG; column `+` shortcut; BoardColumn uses plain elements for drag grip and column name
  - **Plan 05** — `BoardDragContext` detects `col-` prefixed ids for column drag, dispatches `onReorderColumn`; `sortActive` guard blocks same-bucket task reorder; board page wires `SortableContext` (horizontal strategy) + `handleReorderColumn` optimistic patch + sort-active chip

  ## Test plan

  - [ ] All `bun run --filter @future/web-planner test:unit` tests pass
  - [ ] All `bun run --filter @future/api test:unit` tests pass
  - [ ] `bun run --filter "@future/*" build` exits 0
  - [ ] Manual QA checklist in Plan 06 Step 9 completed (progress icons, priority icons, mutation stability, column UI, drag-drop)
  - [ ] Column drag reorder persists after page refresh
  - [ ] Sort-active chip visible when `?sort=*` param is set; same-bucket drag blocked in sort mode
  - [ ] No regressions in `planner-smoke.spec.ts` flows (plan lifecycle, task move, mark complete, assignee)

  🤖 Generated with [Claude Code](https://claude.com/claude-code)
  EOF
  )"
  ```

  Expected: PR URL printed. Copy it and share with reviewers.


  Expected: all checks pass within the CI run. If a check fails, investigate and push a fix commit.
