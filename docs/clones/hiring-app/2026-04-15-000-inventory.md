## Research-Informed Design Decisions

| Decision                    | Choice                                                          | Rationale                                                                                                                                                  |
| --------------------------- | --------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Candidate vs Application    | **Separate entities**                                           | Industry consensus (Greenhouse, Odoo 18, Workday). One person can apply to multiple jobs, get rejected, re-apply. Legacy merges them — known anti-pattern. |
| Job Position vs Job Opening | **Merged into Job Opening** (Lever/Ashby simplified model)      | Appropriate for current scale. Job Opening carries title, description, department, skills, headcount, pipeline. No separate Position entity.               |
| Pipeline stages             | **Configurable per template** (database records, not hardcoded) | Odoo/Greenhouse pattern. `pipeline_template` → `pipeline_stage` (ordered). Each job opening references a template.                                         |
| Interview evaluation        | **Independent scorecards** (Greenhouse pattern)                 | Each interviewer submits independently with per-criterion ratings + overall recommendation. Prevents anchoring bias.                                       |
| Skill taxonomy              | **Shared single source of truth** (new infrastructure)          | Odoo 18 / Workday Skills Cloud pattern. Shared across hiring, people, projects, performance. DDD: owned by one module, consumed by others.                 |
| Compliance                  | **Vietnam PDPD**                                                | Candidate consent tracking, data retention policies, right to deletion/anonymization.                                                                      |
| Offer management            | **Merged into Recruitment Pipeline**                            | Offer is a late pipeline stage with its own entity but same module boundary. Uses Kernel DecisionFacade for approval workflow.                             |
| Finance integration         | **Event-based** (deferred)                                      | Emit events for budget reservation/commitment. Finance module is stub — wire when built.                                                                   |

## Discovered Modules

### Module 0: Shared Skill Taxonomy (Prerequisite)

- **Source location:** `libs/staticData/` (technologies, levels)
- **Description:** Create shared skill/competency taxonomy as single source of truth. Replaces free-text skills in Projects (`projectRole.skillsRequired`), unstructured JSONB in People (`profileSection`), and CRUD tables in legacy hiring (`technologies`, `level`). Owned by Kernel or a dedicated shared package.
- **Detected tech:** Drizzle schema + domain entity + query facade
- **Dependencies:** None (foundational)
- **Size:** Medium (schema + entities + facade + migration of existing free-text data)
- **Priority:** High (prerequisite for hiring, also benefits people + projects)
- **Status:** `pending-refinement`

### Module 1: Candidate Management & Talent Pool

- **Source location:** `libs/candidates/`, `libs/talentPool/`, frontend `pages/candidate/`, `pages/candidate-details/`, `pages/talent-pool/`, `pages/black-list/`
- **Description:** Candidate person records (CRM), multi-channel contacts, activity timeline, CV attachments, tags, talent pool collections, blacklist. Separate from applications. The "who" independent of "which job."
- **Detected tech:** Legacy: raw SQL + CandidatesService (569 LOC) + TalentPoolDataService (281 LOC). Frontend: ~14K LOC candidate details with 6 sub-tabs.
- **Dependencies:** Shared Skill Taxonomy (for candidate skills), Kernel (audit, actors)
- **Size:** Large (~25 legacy files backend, ~50+ frontend components)
- **Priority:** High
- **Status:** `pending-refinement`

### Module 2: Recruitment Pipeline & Offers

- **Source location:** `libs/recruitments/`, frontend `pages/recruitment/`, `pages/recruitment-details/`, `pages/board/`, `pages/candidate-flow/`
- **Description:** Job openings (merged position+opening model), configurable pipeline templates/stages, applications (candidate journey through a job's pipeline), stage transitions (analytics foundation), hiring team roles, rejection reasons, offer entity with approval workflow via Kernel DecisionFacade. Includes Kanban board view and candidate flow visualization.
- **Detected tech:** Legacy: RecruitmentsService (56 LOC — very thin, most logic in frontend), junction table `ref_recruitment_to_candidate`. Frontend: recruitment filter/form, Kanban board (dnd-kit), candidate flow (React Flow/xyflow).
- **Dependencies:** Candidate Management, Shared Skill Taxonomy, Kernel (decisions, departments, audit), Projects (optional `project_role_id` link)
- **Size:** Large (new entities far exceed legacy — pipeline templates, applications, stage transitions, offers, hiring team, job postings are all new)
- **Priority:** High
- **Status:** `pending-refinement`

### Module 3: Interview & Evaluation

- **Source location:** Frontend `pages/interview/`, `pages/calendar-schedule/`, `components/scheduler/`
- **Description:** Interview scheduling, room booking, calendar view, structured scorecards (Greenhouse pattern — independent per-interviewer evaluation with per-criterion ratings + overall recommendation), interview round templates with criteria/questions, panelist assignments, survey/evaluation sending.
- **Detected tech:** Legacy backend: minimal (interview data mostly in Google Forms/Calendar). Frontend: DevExpress Scheduler (~713 LOC), interview filter/table/modal, appointment components.
- **Dependencies:** Recruitment Pipeline (application reference), Candidate Management, Notifications (reminders), Documents (evaluation templates)
- **Size:** Medium (mostly new — legacy has minimal backend for interviews)
- **Priority:** High
- **Status:** `pending-refinement`

### Module 4: Hiring Reference Data

- **Source location:** `libs/staticData/`, `libs/sourceOfCV/`, frontend `pages/static/*` (cv-source, contact-type, interview-room, interview-type, candidate-status)
- **Description:** Hiring-domain-specific master data: candidate statuses, CV sources, contact types, interview rooms, interview types, rejection reasons. Excludes technologies/levels/departments/projects (moved to shared taxonomy or already in Kernel/Projects).
- **Detected tech:** Legacy: StaticDataService (1,052 LOC — largest service, handles 7 entity types). Frontend: generic CRUD forms per entity.
- **Dependencies:** None
- **Size:** Small (reduced scope — only hiring-specific reference data, not shared concepts)
- **Priority:** High (other modules reference this data)
- **Status:** `pending-refinement`

### Module 5: AI & CV Processing

- **Source location:** Frontend `components/batch-cv-upload-modal/`, `components/batch-cv-progress-modal/`, `components/cv-comparison-modal/`, API endpoints `/read-cv/*`
- **Description:** AI-powered CV parsing (extract structured candidate data from uploaded CVs), batch CV upload/processing, candidate-to-job matching with semantic skill comparison, CV comparison view. Registers as agent tools via Agents module framework.

- **Dependencies:** Candidate Management (creates/enriches candidates), Recruitment Pipeline (matching against job requirements), Shared Skill Taxonomy (structured skill extraction), Agents (tool executor + permissions)
- **Size:** Medium
- **Priority:** Medium
- **Status:** `pending-refinement`

### Module 6: Email & Communication

- **Description:** Email template management with merge fields, candidate communication (interview invites, offers, rejections), optional auto-send on stage transitions, sent email tracking/logging.
- **Detected tech:** Legacy: CKEditor 5 rich text editor for templates, styled HTML emails. Target: uses Admin `TenantEmailConfig` for delivery, Documents module for template rendering.

### Module 7: Hiring Analytics & Reporting

- **Source location:** Frontend `pages/reports/`

- **Detected tech:** Legacy: basic reports (described as "inaccurate" by stakeholder). Frontend: Recharts for visualization, PDF export. Target: query layer over hiring data, future integration with Insights module (Athena).

- **Priority:** Medium
- **Status:** `pending-refinement`

- **Dependencies:** Candidate Management, Recruitment Pipeline
- **Size:** Small
- **Priority:** Medium
- **Status:** `pending-refinement`

## Cross-Module Event Flow

```
Projects: StaffingRequestCreated ──→ Hiring: (optional) auto-create job opening
Hiring: OfferAccepted / CandidateHired ──→ People: create employment profile + onboarding
                                         ──→ Projects: create allocation (if project-linked)
                                         ──→ Identity: provision user account
                                         ──→ Finance: confirm headcount cost (when built)
People: OffboardingStarted ──→ Projects: close allocations, reopen roles
                            ──→ Hiring: (optional) auto-create backfill requisition
```
