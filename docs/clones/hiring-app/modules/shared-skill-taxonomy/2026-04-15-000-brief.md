# Module Brief: Shared Skill Taxonomy

> **Module:** 0 — Shared Skill Taxonomy (Prerequisite)
> **Source:** `libs/staticData/` (technologies, levels) in hiring-app-api-nest
> **Target:** `apps/api/src/modules/kernel/` (core schema)
> **Date:** 2026-04-15
> **Status:** Refined

---

## Purpose

Create a single source of truth for skills, skill categories, skill proficiency levels, and seniority levels. Owned by the Kernel module in the `core` schema, consumed by hiring, people, projects, and performance modules via `KernelSkillFacade`.

Replaces:

- Legacy hiring app's flat `technologies` and `level` tables (single FK, no proficiency)
- People module's unstructured JSONB skill payloads (deferred migration)
- Projects module's free-text `skillsRequired` arrays (deferred migration)

## Decisions

| Feature                  | Decision                                                                            | Rationale                                                                                                                           |
| ------------------------ | ----------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| Location                 | **Kernel module** (core schema)                                                     | Follows existing department pattern for shared reference data                                                                       |
| Entity model             | **skill_category → skill → skill_level** (3 tables) + **seniority_level** (1 table) | Industry standard (Odoo 18 / Workday Skills Cloud). Enables proficiency matching.                                                   |
| Seniority vs proficiency | **Separate concepts**                                                               | Skill proficiency = "how good at X" (Beginner→Expert). Seniority = "career stage" (Intern→Principal). Different axes.               |
| Multi-skill support      | **Many-to-many junctions** (owned by consuming modules)                             | Legacy single-FK is too limiting. Each module creates its own junction table referencing Kernel skill IDs.                          |
| Existing data migration  | **Deferred**                                                                        | People JSONB and Projects text[] stay as-is. No blast radius on working modules. Hiring uses taxonomy from day one.                 |
| Access control           | **Existing Kernel RBAC**                                                            | `kernel:skill:read` (all roles), `kernel:skill:manage` (tenant_admin + hr_ops), `kernel:seniority-level:manage` (tenant_admin only) |

## Data Model

### skill_category (core schema)

| Column      | Type      | Constraints                 |
| ----------- | --------- | --------------------------- |
| id          | uuid (v7) | PK                          |
| tenant_id   | uuid      | NOT NULL                    |
| name        | text      | NOT NULL, UNIQUE per tenant |
| description | text      | nullable                    |
| is_active   | boolean   | NOT NULL, default true      |
| created_at  | timestamp | NOT NULL, default now       |
| updated_at  | timestamp | NOT NULL, default now       |

### skill (core schema)

| Column      | Type      | Constraints                                   |
| ----------- | --------- | --------------------------------------------- |
| id          | uuid (v7) | PK                                            |
| tenant_id   | uuid      | NOT NULL                                      |
| category_id | uuid      | NOT NULL, FK → skill_category                 |
| name        | text      | NOT NULL, UNIQUE per (tenant_id, category_id) |
| description | text      | nullable                                      |
| is_active   | boolean   | NOT NULL, default true                        |
| created_at  | timestamp | NOT NULL, default now                         |
| updated_at  | timestamp | NOT NULL, default now                         |

### skill_level (core schema)

| Column      | Type      | Constraints                                           |
| ----------- | --------- | ----------------------------------------------------- |
| id          | uuid (v7) | PK                                                    |
| tenant_id   | uuid      | NOT NULL                                              |
| name        | text      | NOT NULL, UNIQUE per tenant                           |
| rank        | integer   | NOT NULL, UNIQUE per tenant (for ordering/comparison) |
| description | text      | nullable                                              |
| created_at  | timestamp | NOT NULL, default now                                 |
| updated_at  | timestamp | NOT NULL, default now                                 |

### seniority_level (core schema)

| Column      | Type      | Constraints                 |
| ----------- | --------- | --------------------------- |
| id          | uuid (v7) | PK                          |
| tenant_id   | uuid      | NOT NULL                    |
| name        | text      | NOT NULL, UNIQUE per tenant |
| rank        | integer   | NOT NULL, UNIQUE per tenant |
| description | text      | nullable                    |
| is_active   | boolean   | NOT NULL, default true      |
| created_at  | timestamp | NOT NULL, default now       |
| updated_at  | timestamp | NOT NULL, default now       |

## Facade: KernelSkillFacade

Exported from Kernel module. Read-only for most consumers.

### Query methods

- `listSkillCategories(tenantId, activeOnly?)` → SkillCategory[]
- `listSkills(tenantId, categoryId?, activeOnly?)` → Skill[]
- `listSkillLevels(tenantId)` → SkillLevel[]
- `listSeniorityLevels(tenantId, activeOnly?)` → SeniorityLevel[]
- `getSkillById(skillId, tenantId)` → Skill | null
- `getSkillsByIds(skillIds[], tenantId)` → Skill[]
- `getSeniorityLevelById(levelId, tenantId)` → SeniorityLevel | null

## Permission Keys

| Key                             | Roles                | Locked?      |
| ------------------------------- | -------------------- | ------------ |
| `kernel:skill:read`             | All roles            | Yes (locked) |
| `kernel:skill:manage`           | tenant_admin, hr_ops | No           |
| `kernel:seniority-level:manage` | tenant_admin         | No           |

## Dependencies

- None (this is the foundational prerequisite)

## Risks

- **Seed data completeness:** Initial skill catalog will be small. Tenants need admin UI to add their own. The CRUD tRPC procedures cover this.
- **Name normalization:** "React" vs "ReactJS" vs "React.js" — UNIQUE constraint per (tenant, category) prevents exact duplicates. AI-powered dedup can be added later.
