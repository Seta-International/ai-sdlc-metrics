# Clone Brief: holidays

## Business Purpose

Company-wide holiday management for the `time` module. Holidays mark date ranges as non-working days and automatically link to existing `time_sheet` rows so that attendance logic (late/lack calculations) correctly treats those days as holidays. Full CRUD is restricted to the `hrm` role.

---

## Source Analysis

- **Key files:**
  - `server/services/admin.service.js` — `createHolidayService`, `updateHoliday`, `getAllHolidaysSevice`, `deleteHolidaysSevice`, `checkDuplicateHoliday`
  - `server/query/admin.query.js` — `checkDuplicateHolidayQuery`, `insertHolidayQuery`, `updateHolidayIdInTimesheetQuery`, `getAllHolidaysQuery`, `deleteHolidayQuery`, `updateHoliday`, `deleteHolidayIdInTimesheet`
  - `server/routes/admin.js` — 4 REST routes: GET/POST `/holiday`, PUT/DELETE `/holiday/:id`
  - `server/helper/calculation.js` — `countOffdays` (pure date arithmetic, no external deps)

- **Dependencies:**
  - `time_sheet` table (same DB, for `holiday_id` back-linking) — within the same `time` module boundary
  - No cross-module reads needed

- **External integrations:** None

## Bugs and Issues Found in Source

| #      | Issue                                                                                                                                                                                                                                                                                                                        | Severity | Fix                                                                                                                                                                                         |
| ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| BUG-01 | `updateHoliday` never calls `checkDuplicateHoliday` — updates can create overlapping holidays                                                                                                                                                                                                                                | High     | Add overlap check on update (exclude self via `id`)                                                                                                                                         |
| BUG-02 | `deleteHolidayQuery` uses raw string interpolation with `id` (SQL injection risk)                                                                                                                                                                                                                                            | High     | Use Drizzle parameterized queries                                                                                                                                                           |
| BUG-03 | `getAllHolidaysQuery` has no tenant filter — returns holidays for all tenants                                                                                                                                                                                                                                                | Critical | Add `tenant_id = :tenantId` filter everywhere                                                                                                                                               |
| BUG-04 | `countOffdays` uses `Math.round`, but description says "ceiling". Actual behavior: round half-up. For a 1-day holiday where start = end, result = 1 (correct due to `                                                                                                                                                        |          | 1`fallback). For ranges, round vs ceil only differs at .5 — unlikely in whole-day ranges. **Preserve`Math.round` behavior exactly\*\* to avoid changing stored durations for existing rows. |
| BUG-05 | `updateHoliday` runs UPDATE first, then clears old `holiday_id` refs, then re-links. If the holiday row update changes `start_date` but the old `holiday_id` clear uses `holidayId` from the updated row — this is correct. But if update fails after clear, refs are orphaned until ROLLBACK. Transaction covers this — OK. |

## Feature Decisions

| Feature                                            | Decision             | Rationale                                                                                                                                                                |
| -------------------------------------------------- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `holiday` table schema                             | **keep + reimagine** | Port to `pgSchema('time')`, add `id` (uuid v7), `tenant_id` (uuid notNull). Column names stay the same (snake_case). `start_date` as `date` type, `duration` as integer. |
| `Holiday` domain entity                            | **keep**             | Simple value object: id, tenantId, name, startDate, duration, description. Add computed `endDate` property.                                                              |
| `countOffdays` duration calculation                | **keep**             | Pure date math; `Math.round` behavior must be preserved exactly. Port as static method on `HolidayDuration` value object.                                                |
| List holidays (GET /holiday)                       | **keep**             | Port to tRPC query. Add `endDate` as computed field (start_date + duration - 1). Scoped by `tenant_id`.                                                                  |
| Create holiday with overlap check + timesheet link | **keep + reimagine** | Port transaction logic. Fix BUG-01 omission: overlap check on update. Use Drizzle `daterange` overlap via `sql` template.                                                |
| Update holiday: clear old refs, re-link            | **keep**             | Transaction pattern is correct. Add missing overlap check (BUG-01).                                                                                                      |
| Delete holiday: clear refs first, then delete      | **keep**             | Correct order — prevents orphaned `holiday_id` in timesheet.                                                                                                             |
| Overlap check via PostgreSQL `daterange &&`        | **keep**             | Correct and efficient. Port using Drizzle `sql` template literal.                                                                                                        |
| `roleAndPermission(['admin'])` middleware          | **reimagine**        | Replace with tRPC `hrmProcedure` (maps legacy `admin` role to `hrm` in Future). Staff and manager get no access.                                                         |
| Raw `yesql` SQL                                    | **skip**             | Drizzle ORM replaces all raw SQL.                                                                                                                                        |
| No tenant isolation                                | **fix** (critical)   | Every query must include `tenant_id` filter. The `holiday` table must have `tenant_id` column (BUG-03).                                                                  |
| SQL injection in delete                            | **fix**              | Drizzle parameterized queries eliminate this (BUG-02).                                                                                                                   |
| Missing overlap check on update                    | **fix**              | Add it (BUG-01).                                                                                                                                                         |

## Key Business Rules (must be preserved)

| Rule                                                  | Implementation note                                                                                                                                                                         |
| ----------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Holidays are date-range based: start_date + duration  | `duration = countOffdays(startDate, endDate) \|\| 1`. `endDate` is inclusive.                                                                                                               |
| No overlapping holidays                               | PostgreSQL `daterange(:startDate::date, :endDate::date, '[]') && daterange(h.start_date::date, (h.start_date::date + h.duration - 1)::date, '[]')`. Exclude self on update via `id != :id`. |
| Create: update timesheet rows in same transaction     | `UPDATE time.time_sheet SET holiday_id = :id WHERE date::date BETWEEN :startDate AND :endDate AND tenant_id = :tenantId`                                                                    |
| Update: clear old refs, then re-link, in transaction  | Clear: `UPDATE time.time_sheet SET holiday_id = NULL WHERE holiday_id = :id AND tenant_id = :tenantId`. Then re-link with new range.                                                        |
| Delete: clear refs first, then delete, in transaction | Same clear pattern before `DELETE FROM time.holiday WHERE id = :id AND tenant_id = :tenantId`                                                                                               |
| hrm only                                              | All procedures use `hrmProcedure` context (role check in tRPC middleware)                                                                                                                   |
| endDate on list                                       | Computed as `start_date + duration - 1` days, formatted as `yyyy-mm-dd`                                                                                                                     |

## Risks & Assumptions

- The `time.time_sheet` table will be defined in `infrastructure/schema/time.schema.ts` (by the `attendance` task). The `holidays` task depends on the schema definition existing. However, the Drizzle schema reference is only needed at build time — the task can reference the table name via `sql` template literals if the schema definition isn't merged yet.
- `countOffdays` `Math.round` vs `Math.ceil` discrepancy with the user brief description. **Decision: use `Math.round` to match exact legacy behavior.** Document the discrepancy in code.
- No UI task included in this brief — the `HolidaySetting` React page is a frontend concern for the `web-time` zone. This will be a separate clone task (not in scope for this single-task refinement).
