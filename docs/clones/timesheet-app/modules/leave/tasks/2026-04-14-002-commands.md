# Task: commands

## Scope

Implement all write-side command handlers for the leave module, their Drizzle repository implementations, and register everything in `TimeModule`. Covers:

1. `SubmitLeaveRequestCommand` — staff submits a leave request
2. `CancelLeaveRequestCommand` — staff cancels their own `new` request
3. `ConfirmLeaveRequestCommand` — manager sets `approve_by` (confirms)
4. `ApproveLeaveRequestCommand` — HRM approves; deducts balance
5. `RejectLeaveRequestCommand` — HRM rejects; no balance change
6. `UpdateLeaveBalanceCommand` — HRM manually overrides a user's balance
7. `RecalculateAnnualLeaveCommand` — HRM triggers year-end recalculation for all (or subset of) users
8. `CreateLeaveReasonCommand` / `UpdateLeaveReasonCommand` / `DeleteLeaveReasonCommand` — HRM manages reasons

Also includes the three Drizzle repository implementations (the concrete classes behind the interfaces from Task 001).

## Business Context

These are the operations that change state. The most critical invariants:

- BR-04: Carry-over requests must start AND end within Jan–March
- BR-05: `off_time_hour` is calculated via `OffTimeHourCalculator` using the user's work schedule fetched from `WorkTimeQueryFacade`
- BR-06: Balance is deducted **only** when HRM sets `status = 'approved'`, not at submission
- BR-11: Manager view uses only `manager_id` (no `line_manager_id`)

The recalculation logic (`calTotalTimeUserLeaveByYear`) needs hired_date and max_leave_day from the people module via `PeopleQueryFacade`.

- **Key source logic to port:**

  **Submit leave request (`postLeaveRequestService`):**
  1. Validate: `startDateTime`, `endDateTime`, `reasonId` required
  2. Check for overlapping request (same user, same `reasonId`, overlapping date range, status != 'rejected') → 409 if found
  3. If `leaveCarryOverType`: check month of start AND end ≤ March → 404 if outside Q1
  4. Calculate `offTimeHour` via `getOffTimeHourService` → uses work schedule segments
  5. Check balance: if `carry_over` type → `carryOverRemain >= offTimeHour`; if `annual` → `totalRemain >= offTimeHour` → 500 if insufficient
  6. Insert `request` row with `status = 'new'`

  **Approval / balance deduction (`updateUserLeaveWhenConfirmRequest`):**
  - Only fires when `status = 'approved'`
  - Fetches `user_leave` for user + year of `start_date_time`
  - If `annual`: `total_remain -= off_time_hour`; validates `total_remain >= off_time_hour`
  - If `carry_over`: `carry_over_remain -= off_time_hour`; validates `carry_over_remain >= off_time_hour`
  - Runs in same transaction as status update

  **Year-end recalculation (`calTotalTimeUserLeaveByYear`):**
  - For each active user:
    - Sum approved+confirmed `off_time_hour` for current year (annual leave reason) → `time_leave_now`
    - Sum approved+confirmed `off_time_hour` for current year (carry-over reason) → `carry_over_used`
    - Sum approved+confirmed `off_time_hour` for _last_ year (annual leave reason) → `time_leave_last_year`
    - Compute `carry_over_h = last_year_allocation - last_year_used` (hours not used last year)
    - Cap carry_over by `MAX_LEAVE_CARRY_OVER * 8` hours (env var); if `overThreeMonth = true`, carry_over = 0
    - For users hired in their first year: prorate `totalLeave` by remaining months
    - Upsert `user_leave` row for current year
  - **Reimagine:** replace O(n²) nested map with a join query; replace env var `MAX_LEAVE_CARRY_OVER` with a tenant-level config (from `AdminQueryFacade` or hardcode 5 days = 40h as default, configurable via command parameter)

## Data Model

No new tables. Operations on tables from Task 001.

**Balance deduction transaction pattern (approve):**

```
db.transaction(async (tx) => {
  // 1. Update request status
  await leaveRequestRepo.update(id, tenantId, { status: 'approved', approvedBy: hrmActorId }, tx)
  // 2. Fetch balance
  const balance = await leaveBalanceRepo.findByActorAndYear(actorId, tenantId, year, tx)
  if (!balance) throw new LeaveBalanceNotFoundException(actorId, year)
  // 3. Deduct
  if (leaveType === 'annual') {
    if (balance.totalRemain < offTimeHour) throw new InsufficientLeaveBalanceException()
    await leaveBalanceRepo.update(balance.id, tenantId, { totalRemain: balance.totalRemain - offTimeHour }, tx)
  } else {
    if (balance.carryOverRemain < offTimeHour) throw new InsufficientLeaveBalanceException()
    await leaveBalanceRepo.update(balance.id, tenantId, { carryOverRemain: balance.carryOverRemain - offTimeHour }, tx)
  }
})
```

## Edge Cases

- **Submit:** if no `leave_balance` row exists for the user+year → throw `LeaveBalanceNotFoundException` with helpful message ("No leave balance found for year X — contact HRM")
- **Submit carry-over:** check `new Date(startAt).getMonth() + 1 <= 3` AND `new Date(endAt).getMonth() + 1 <= 3` — both must be in Q1
- **Submit:** if `offTimeHour <= 0` (e.g. date range is entirely weekend) → throw `InvalidLeaveRequestDateRangeException`
- **Cancel:** only allowed when `status === 'new'`; throw `LeaveRequestNotCancellableException` otherwise
- **Confirm:** must verify requester is the `manager_id` of the request owner (via `PeopleQueryFacade`)
- **Approve:** if `status` is not `'confirmed'` → throw `InvalidLeaveStatusTransitionException` (HRM must not skip manager confirmation step)
- **Reject:** allowed from `'new'` or `'confirmed'` — reject at any stage before approval
- **Recalculate:** users hired in the same year as `year` → skip (no carry-over for brand-new hires); users hired in `year - 1` → prorate
- **Recalculate:** if `actorIds` is provided, only recalculate those users (HRM can target specific people)
