# Clone Brief: leave

## Business Purpose

Manages annual leave and carry-over leave for employees:

- Staff submit leave requests (annual or carry-over); HRM approves or rejects; manager confirms (sets `approve_by`)
- Leave balance (hours) is tracked per user per year in `user_leave`; deducted only on HRM approval (BR-06)
- Carry-over leave is valid Jan–March only; requests outside Q1 are rejected (BR-04)
- Leave is measured in hours: 8h = 1 working day; `off_time_hour` is derived from the user's work schedule (BR-05)
- HRM can view all balances, manually override them, and trigger year-end recalculation

## Source Analysis

- **Key files:**
  - `server/services/userLeave.service.js` — balance CRUD, recalculation logic
  - `server/query/userLeave.query.js` — all SQL for `user_leave` table
  - `server/services/request.service.js` — `postLeaveRequestService`, `getOffTimeHourService`, `updateMultipleRequest`, `getMyRequestService`, `getRequestForManagerService`, `getRequestsForAdmin`
  - `server/query/request.query.js` — `postLeaveRequestQuery`, `checkDuplicateLeaveRequestQuery`, `selectWorkTimeQuery`, leave-filtered variants of `getRequestsQuery`
  - `server/helper/calculation.js` — `getOffTimeHour` and all supporting functions (pure math)
  - `server/helper/constants.js` — hardcoded discriminator UUIDs

- **Dependencies:**
  - `work_time` table (from work-time module) — needed to compute `off_time_hour`; accessed via cross-module query in legacy (direct SQL join). In Future: `WorkTimeQueryFacade`
  - `user` table (people module) — needed for `hired_date`, `max_leave_day`, `name`, `manager_id` in recalculation. In Future: `PeopleQueryFacade`

- **External integrations:** Email (Nodemailer) on approval/rejection — **skip**; Future uses outbox events

## Decisions

| Feature                                                     | Decision      | Rationale                                                                                                                                                                                           |
| ----------------------------------------------------------- | ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------- |
| `off_time_hour` calculation (`calculation.js`)              | **keep**      | Pure math functions, well-defined algorithm, business-critical; port faithfully as a pure TypeScript domain service                                                                                 |
| Leave request submission with duplicate check + quota check | **keep**      | Core business flow; carry forward all guard logic                                                                                                                                                   |
| Carry-over Q1 restriction (BR-04)                           | **keep**      | Explicit business rule; enforce in domain                                                                                                                                                           |
| Balance deduction on approval only (BR-06)                  | **keep**      | Core invariant; must be enforced in `ApproveLeaveRequestHandler`, not at submission time                                                                                                            |
| Two-step approval (manager confirm → HRM approve)           | **keep**      | Business process; `approve_by` (manager confirm) and `status=approved` (HRM) are separate operations                                                                                                |
| Year-end recalculation (`calTotalTimeUserLeaveByYear`)      | **reimagine** | The source uses a PG stored procedure (`updateOrInsertUserLeave`) and mixes carry-over logic poorly (nested map-over-map is O(n²)). Reimagine as a proper command with transactional Drizzle upsert |
| `line_manager_id` in manager view                           | **skip**      | BR-11: single `manager_id` only; `line_manager_id` is gone                                                                                                                                          |
| Email notifications                                         | **skip**      | Future uses outbox events; email is a downstream subscriber concern                                                                                                                                 |
| `managerOnly` flag in request listing                       | **reimagine** | Was a band-aid for `line_manager_id` complexity; simplified to "manager sees their direct reports"                                                                                                  |
| Raw SQL string interpolation                                | **reimagine** | Security bug (SQL injection); replace with Drizzle parameterized queries                                                                                                                            |
| Hardcoded UUIDs as discriminators                           | **reimagine** | `leaveRequestTypeId`, `reasonLeaveRequestId`, `leaveCarryOverType` become an enum column `leave_type` on the `leave_request` table: `'annual'                                                       | 'carry_over'` |
| `reason` table (leave reasons with `max_request_day`)       | **keep**      | Business reference data; HRM manages it; staff/manager read it                                                                                                                                      |
| Leave balance view + manual override (HRM)                  | **keep**      | Operational need; add proper role guard (legacy had none)                                                                                                                                           |

## Architecture Mapping

| Source pattern                            | Target pattern                                                         |
| ----------------------------------------- | ---------------------------------------------------------------------- |
| Express route handler                     | tRPC procedure (role-checked via `KernelDecisionFacade`)               |
| Raw SQL in `.query.js`                    | Drizzle ORM in `infrastructure/repositories/`                          |
| `req.user.id` from cookie                 | `ctx.actorId` + `tenantId` from tRPC context                           |
| `req.postgres.query()`                    | Injected repository via NestJS DI                                      |
| `camelCaseData()` utility                 | Drizzle returns camelCase by default                                   |
| `user.title` role check (none in legacy!) | `KernelDecisionFacade.can(actorId, permission, tenantId)`              |
| `updateOrInsertUserLeave` stored proc     | Drizzle `insert(...).onConflictDoUpdate(...)`                          |
| `u.max_leave_day`, `u.hired_date`         | `PeopleQueryFacade.getEmploymentProfile(actorId, tenantId)`            |
| `work_time` direct SQL join               | `WorkTimeQueryFacade.getScheduleForUser(actorId, tenantId, dateRange)` |
| `getOffTimeHour` pure JS function         | `OffTimeHourCalculator` pure TS domain service (no DI needed)          |

## Risks & Assumptions

- **Cross-module data for recalculation:** The recalculation needs `hired_date` and `max_leave_day` from the people module. These must come via `PeopleQueryFacade`. If that facade doesn't yet expose those fields, the recalculation task will need a stub.
- **Work schedule for `off_time_hour`:** Needs `WorkTimeQueryFacade.getSchedulesForUser(actorId, tenantId, fromDate, toDate)` returning work time rows. If the work-time module isn't built yet, provide an interface stub and note the dependency.
- **Tenant isolation:** Every table row must have `tenant_id`. The leave module owns `leave_request`, `leave_balance`, `leave_reason`. All queries must filter by `tenant_id`.
- **`max_leave_day` ownership:** In legacy this lives on the `user` table. In Future it belongs to `people` module. The recalculation query must use `PeopleQueryFacade`; the leave module must not query the `people` schema directly.
- **Status transitions:** Valid transitions for a leave request: `new → confirmed` (manager sets `approve_by`) → `approved | rejected` (HRM). Cancel is allowed from `new` by the requester.
