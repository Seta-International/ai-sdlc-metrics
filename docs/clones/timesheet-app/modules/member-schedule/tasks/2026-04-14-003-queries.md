## Business Context

### Registration Status Check

Before a manager can tell their team to check in, they want to know: "How many of my team members haven't registered a work schedule yet?" and "Have I registered my own schedule?" This drives a UI badge/warning in `web-time`.

In the legacy code (`checkUserHasRegisterSchedule`):

- `hasMemberNotRegister`: count of team members (users where `manager_id = requesterId`) who have no active schedule. The legacy query does a LEFT JOIN on `public.user` and filters `schedule IS NULL`. This cross-schema join must be replaced with `PeopleQueryFacade`.
- `checkCurrentUserNotRegister`: boolean — `true` if the requesting user has NO active schedule (i.e., they need to register).

### Internal: getActiveMemberSchedule

The attendance module needs to check the user's active RRule schedule during online check-in (BR-08). Since member-schedule and attendance are both sub-domains of the `time` module (same NestJS module boundary), attendance can call `TimeQueryFacade` methods directly — no cross-module boundary is crossed.

This method must be fast (on the check-in critical path). It should hit the indexed `(tenant_id, actor_id, status)` column.

## Source Reference

### `checkUserHasRegisterSchedule` (memberSchedule.service.js, lines 38–55)

```js
async function checkUserHasRegisterSchedule(req) {
  const userId = req.user.id
  const hasMemberNotRegister = await req.postgres.query(
    getListMemberScheduleByManager({ userId, ...req.query }),
  )
  const checkCurrentUserNotRegister = await req.postgres.query(
    getMemberSchedule({ userId, ...req.query }),
  )
  return {
    code: 200,
    data: {
      hasMemberNotRegister: hasMemberNotRegister?.rowCount || 0,
      checkCurrentUserNotRegister: checkCurrentUserNotRegister?.rowCount === 0,
    },
  }
}
```

### `getListMemberScheduleByManager` (memberSchedule.query.js)

```sql
SELECT *
FROM "user" u
LEFT JOIN work_schedule ws ON u.id = ws.user_id AND state = 'ACTIVE'
WHERE u.manager_id = :userId AND u.status = 'active' AND schedule IS NULL
```

This query:

1. Gets active employees under the requesting manager (`u.manager_id = :userId AND u.status = 'active'`)
2. LEFT JOINs their schedule
3. Filters to those with no active schedule (`schedule IS NULL`)

Replacement strategy in Future:

- Step 1: `PeopleQueryFacade.getDirectReports(requesterId, tenantId)` → returns list of `actorId`
- Step 2: `repo.findActiveStatusForUsers(actorIds, tenantId)` → returns `{ actorId, hasActive }[]`
- Count how many have `hasActive = false`

## Handler: GetScheduleRegistrationStatusHandler

```
@QueryHandler(GetScheduleRegistrationStatusQuery)
export class GetScheduleRegistrationStatusHandler
  implements IQueryHandler<GetScheduleRegistrationStatusQuery, ScheduleRegistrationStatusResult>
```

**Algorithm:**

```
1. Fetch self's active schedule: repo.findActive(requesterId, tenantId)
   selfUnregistered = (result === null)

2. Fetch direct reports: PeopleQueryFacade.getDirectReports(requesterId, tenantId)
   → returns actorId[] (active employees under this manager)
   If empty (staff has no reports, or hrm calling): teamMembersUnregistered = 0

3. repo.findActiveStatusForUsers(directReportIds, tenantId)
   → returns { actorId, hasActive }[]
   teamMembersUnregistered = count where hasActive === false

4. Return { teamMembersUnregistered, selfUnregistered }
```

**Note on PeopleQueryFacade:** the current `PeopleQueryFacade` in `apps/api/src/modules/people/application/facades/people-query.facade.ts` does not yet have a `getDirectReports` method. You will need to add it. This is a cross-module facade call:

- Add `GetDirectReportsQuery` in `people/application/queries/`
- Add `GetDirectReportsHandler` that queries `people.employment_profile` or the relevant table for `manager_id = requesterId`
- Add `getDirectReports(managerId: string, tenantId: string): Promise<string[]>` to `PeopleQueryFacade`

If `getDirectReports` has already been added as part of another task (e.g., the attendance module task), use the existing implementation. Do not duplicate.
