## Scope

Implement the two write operations for member schedules:

1. `AssignMemberScheduleCommand` + handler — assign an RRule schedule to one or more users (deactivates existing, inserts new, in one transaction per user, full batch atomic)
2. `DeactivateMemberScheduleCommand` + handler — deactivate active schedules for one or more users (no new schedule inserted)
3. tRPC mutations on `timeRouter` for both commands
4. RRule validation helper
5. Unit tests (happy path + error paths) for both handlers

---

## Business Context

The assign operation is the main way HRM/managers set up office schedules for employees. The deactivate operation is used during offboarding or schedule removal. Both accept an array of userIds so the UI can batch-assign/deactivate a team in one action.

**Critical invariant:** a user must never have more than one `status = 'active'` schedule row at a time. The assign operation maintains this by deactivating first, then inserting — both within the same database transaction per user.

**Batch atomicity:** the legacy code ran `Promise.all` per user with no coordination. If user 3 of 5 failed, users 1–2 were already committed. Fix: wrap the entire batch in a single transaction. All users succeed or all roll back.

**RRule validation (new):** the legacy code stored whatever string the client sent. An invalid RRule causes a crash in `checkInOnlineService`. Validate the string before persisting.

**Role-based scoping (new):** the legacy code had no scoping — any authenticated user could assign a schedule to any userId. The new system enforces:

- `staff`: can only assign/deactivate for their own `actorId`
- `manager`: can only target users who are their direct reports (via `PeopleQueryFacade.getDirectReports`)
- `hrm`: can target any user in the tenant

## RRule Validation Helper

```typescript
// domain/value-objects/rrule-schedule.ts
import { RRule } from 'rrule'

export class InvalidScheduleError extends Error {
  constructor(schedule: string, cause?: unknown) {
    super(`Invalid RRule schedule string: "${schedule}"`)
    this.name = 'InvalidScheduleError'
  }
}

/**
 * Validates and normalizes an RRule schedule string.
 * Throws InvalidScheduleError if the string cannot be parsed.
 */
export function parseRRuleSchedule(schedule: string): string {
  try {
    RRule.parseString(schedule)
    return schedule
  } catch (err) {
    throw new InvalidScheduleError(schedule, err)
  }
}

/**
 * Returns true if the given date falls within the schedule (i.e., the user
 * is expected in the office on that date).
 *
 * Used by attendance module for BR-08 check-in validation.
 * inAccordance = false when isScheduledDay returns true (user should be in office).
 */
export function isScheduledDay(scheduleString: string, date: Date): boolean {
  const rule = new RRule(RRule.parseString(scheduleString))
  const startOfDay = new Date(
    Date.UTC(date.getUTCFullYear(), date.getUTCMonth(), date.getUTCDate()),
  )
  const startOfNextDay = new Date(
    Date.UTC(date.getUTCFullYear(), date.getUTCMonth(), date.getUTCDate() + 1),
  )
  const occurrences = rule.between(startOfDay, startOfNextDay)
  return occurrences.length > 0
}
```

Note: `rrule` must be available as a dependency. Check `apps/api/package.json`; if absent, add with `bun add rrule` from `apps/api`.

## Handler: AssignMemberScheduleHandler

```
@CommandHandler(AssignMemberScheduleCommand)
export class AssignMemberScheduleHandler implements ICommandHandler<AssignMemberScheduleCommand>
```

**Algorithm:**

```
1. Validate `schedule` string with `parseRRuleSchedule()` — throw InvalidScheduleError if invalid
2. Enforce role-based scoping:
   - staff: actorIds must equal [requesterId]; throw ForbiddenError otherwise
   - manager: fetch directReports = PeopleQueryFacade.getDirectReports(requesterId, tenantId)
               actorIds must be a subset of directReports; throw ForbiddenError for any out-of-scope id
   - hrm: no restriction (any actorId in tenant)
3. In a single DB transaction:
   For each actorId in actorIds:
     a. repo.deactivateAllForUser(actorId, tenantId, now)
     b. repo.insert({ tenantId, actorId, schedule, status: 'active', activatedAt: now, deactivatedAt: null })
4. Return array of { actorId, scheduleId } for each assigned user
```

**Errors to handle:**

- `InvalidScheduleError` → surface as tRPC `BAD_REQUEST`
- `ForbiddenError` → surface as tRPC `FORBIDDEN`

**Note on transactions:** look at how other handlers in this codebase use Drizzle transactions. Check `apps/api/src/common` for a `DrizzleTransactionManager` or look at existing command handlers that do multi-step writes.

## Handler: DeactivateMemberScheduleHandler

```
@CommandHandler(DeactivateMemberScheduleCommand)
export class DeactivateMemberScheduleHandler implements ICommandHandler<DeactivateMemberScheduleCommand>
```

**Algorithm:**

```
1. Enforce role-based scoping (same logic as AssignMemberScheduleHandler)
2. In a single DB transaction:
   For each actorId in actorIds:
     a. repo.deactivateAllForUser(actorId, tenantId, now)
3. Return count of users processed
```

Deactivation is idempotent: if a user has no active schedule, `deactivateAllForUser` returns 0 rows affected — not an error.

## Unit Tests

### `assign-member-schedule.handler.spec.ts`

- Happy path (hrm assigns to 3 users): deactivates existing + inserts new for each; returns result
- Happy path (staff assigns to self): succeeds
