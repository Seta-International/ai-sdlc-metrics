**Target:** `apps/api/src/modules/time/` (member-schedule sub-domain)
**Date:** 2026-04-14
**Status:** refined

---

## What This Module Does

The member-schedule module owns RRule-based work-schedule patterns per employee. An RRule string (e.g. `FREQ=WEEKLY;BYDAY=MO,TU,WE,TH,FR`) encodes which days a user is expected to be in the office. This schedule is consulted at online check-in time (BR-08): if today falls within the user's office schedule and they check in online, the attendance row is flagged `in_accordance = false` and a violation notification is triggered.

Core operations:

1. **Assign** — deactivate any existing active schedule, insert a new active one (atomic per user; accepted as a batch of userIds).
2. **Deactivate** — set an active schedule to inactive without inserting a new one (soft delete pattern; also batch).
3. **Check registration status** — for a manager: count of direct reports without an active schedule + whether the requesting manager themselves has an active schedule.
4. **Get active schedule** — internal read used by the attendance module for check-in validation; not a public tRPC endpoint.

## Source Discrepancies (fixed in migration)

| Issue                              | Source behavior                                                                        | Fix                                                              |
| ---------------------------------- | -------------------------------------------------------------------------------------- | ---------------------------------------------------------------- |
| Table name                         | Query uses `work_schedule`, route docs say `member_schedule`                           | Target table: `time.member_schedule`                             |
| Column naming                      | `state` with `'ACTIVE'`/`'INACTIVE'`                                                   | Rename to `status` with `'active'`/`'inactive'` (inventory spec) |
| No `tenant_id`                     | Missing from legacy                                                                    | Required on every row in Future                                  |
| No RRule validation                | Invalid string silently stored; crashes at parse time                                  | Validate with `RRule.parseString()` in command; throw on failure |
| Parallel batch without transaction | `Promise.all` per user with no coordination; partial failures leave inconsistent state | Single DB transaction for the entire batch                       |
| Cross-schema `user` JOIN           | `getListMemberScheduleByManager` JOINs public `user` table directly                    | Replace with `PeopleQueryFacade.getDirectReports()`              |
| `deleteWorkSchedule` misnaming     | Route/function called "delete" but only deactivates                                    | Rename to `DeactivateMemberScheduleCommand`                      |

## Feature Decisions

| Feature                             | Decision             | Rationale                                                                                                        |
| ----------------------------------- | -------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `time.member_schedule` table        | **keep + reimagine** | Rename columns, add `tenant_id` + uuid v7 `id`, rename `state` → `status`, add `activated_at` / `deactivated_at` |
| Assign schedule to user(s)          | **keep + reimagine** | Core logic correct. Port to `AssignMemberScheduleCommand`; accept array of userIds; single transaction for batch |
| Deactivate schedule for user(s)     | **keep + reimagine** | Port to `DeactivateMemberScheduleCommand`; rename from "delete"; single transaction                              |
| Check registration status           | **keep + reimagine** | Port to query handler; replace cross-schema `user` JOIN with `PeopleQueryFacade.getDirectReports()`              |
| Get active schedule (internal)      | **keep + reimagine** | Expose as method on `TimeQueryFacade`; attendance module calls this within the same module boundary              |
| RRule validation                    | **fix (add)**        | Validate at command boundary; throw `InvalidScheduleError` for unparseable strings                               |
| Batch transaction safety            | **fix**              | Single transaction per batch; all-or-nothing semantics                                                           |
| `from_date` / `to_date` audit trail | **keep**             | Rename to `activatedAt` / `deactivatedAt`; nullable timestamps; useful for auditing when schedules changed       |
| `moment` dependency                 | **skip**             | Not used in this module                                                                                          |
| `yesql` raw SQL                     | **skip**             | Replaced by Drizzle ORM                                                                                          |
| Cross-schema `user` JOIN            | **skip**             | Use `PeopleQueryFacade` instead                                                                                  |
| Email/FCM notification on violation | **skip**             | Responsibility of attendance module; it consumes the active schedule and emits its own domain events             |

## Key Business Rules (must be preserved)

| Rule                                   | Implementation note                                                                                                                                           |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| One active schedule per user at a time | Before inserting, update all active rows for the user to `status = 'inactive'` in the same transaction                                                        |
| Soft delete only                       | `status = 'inactive'` is the only removal mechanism; no hard deletes                                                                                          |
| BR-08: RRule check-in validation       | `inAccordance = rrule.between(startOfDay, startOfNextDay).length === 0` — today in schedule means user should be in office, so online check-in is a violation |
| Batch atomicity                        | All userIds in a batch must succeed or all roll back                                                                                                          |

## Role Permissions

| Operation                 |     staff     | manager  |   hrm   |
| ------------------------- | :-----------: | :------: | :-----: |
| Assign schedule           | ✓ (self only) | ✓ (team) | ✓ (any) |
| Deactivate schedule       | ✓ (self only) | ✓ (team) | ✓ (any) |
| Check registration status |       ✓       |    ✓     |    ✓    |

Note: The legacy code does not enforce role-based scoping on assign/deactivate (any authenticated user can target any userId array). The target must enforce: staff can only target their own actorId, manager can only target their direct reports, hrm can target any user in the tenant.
