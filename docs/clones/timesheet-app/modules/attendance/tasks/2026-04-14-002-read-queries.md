## Scope

Implement the four read paths for timesheet data:

1. `GetMyTimesheetHandler` — staff's own timesheet (paginated, date-range)
2. `GetMemberTimesheetHandler` — manager's team timesheet (paginated, filtered by badge number)
3. `GetAllTimesheetHandler` — company-wide timesheet (manager + hrm, paginated, filterable by badge/name)
4. `GetTimesheetEntryHandler` — single entry by userId + date
5. `DrizzleTimesheetEntryRepository` — Drizzle implementation of `ITimesheetEntryRepository` (read methods only; write methods added in Task 004)
6. tRPC procedures for all four queries in `interface/trpc/time.router.ts`

---

## Business Context

The timesheet views are the primary daily-use feature. Managers use the member view to spot tardiness. HRM uses the all-view for payroll and compliance. The calendar-row pattern (generate a row for every day in range, even if no check-in exists) is intentional — it shows gaps clearly. This behavior must be preserved.

Cross-module constraint: the member and all-timesheet views need user data (name, badge_number, manager_id). These come from `PeopleQueryFacade`, not direct DB joins.

## Query Shapes

### GetMyTimesheetQuery

```ts
export class GetMyTimesheetQuery {
  constructor(
    public readonly tenantId: string,
    public readonly actorId: string, // the requesting user's actorId
    public readonly fromDate: string, // YYYY-MM-DD
    public readonly toDate: string, // YYYY-MM-DD
    public readonly limit: number,
    public readonly offset: number,
  ) {}
}
```

**Handler behavior:**

1. Validate `fromDate <= toDate`; max range 93 days (one quarter)
2. Call `repo.listByUser({ tenantId, userId: actorId, fromDate, toDate, limit, offset })`
3. Return `{ data: TimesheetCalendarRow[], total: number, page: number }`

### GetMemberTimesheetQuery

```ts
export class GetMemberTimesheetQuery {
  constructor(
    public readonly tenantId: string,
    public readonly managerId: string,
    public readonly fromDate: string,
    public readonly toDate: string,
    public readonly badgeNumber?: string,
    public readonly limit: number = 20,
    public readonly offset: number = 0,
  ) {}
}
```

**Handler behavior:**

1. Validate date range (max 93 days)
2. If `badgeNumber` provided: call `PeopleQueryFacade.findEmployeeByBadge(badgeNumber, tenantId)` to get userId; validate they report to managerId
3. Otherwise: get direct report IDs via `PeopleQueryFacade.getDirectReports(managerId, tenantId)`
4. Call `repo.listByManager({ tenantId, managerId, fromDate, toDate, badgeNumber, limit, offset })`

**Note on repository implementation:** The `listByManager` query in Drizzle needs the user IDs from PeopleQueryFacade to filter without cross-schema joins. Pass `directReportIds: string[]` to the repository method (adjust interface in Task 001 if needed — acceptable since this task comes right after).

### GetAllTimesheetQuery

```ts
export class GetAllTimesheetQuery {
  constructor(
    public readonly tenantId: string,
    public readonly fromDate: string,
    public readonly toDate: string,
    public readonly badgeNumber?: string,
    public readonly name?: string,
    public readonly limit: number = 20,
    public readonly offset: number = 0,
  ) {}
}
```

**Handler behavior:**

1. Validate date range (max 93 days)
2. If `name` filter provided: get matching actor IDs via `PeopleQueryFacade.searchEmployees(name, tenantId)` — never interpolate name into SQL
3. Call `repo.listAll({ tenantId, fromDate, toDate, badgeNumber, actorIds?, limit, offset })`

### GetTimesheetEntryQuery

```ts
export class GetTimesheetEntryQuery {
  constructor(
    public readonly tenantId: string,
    public readonly userId: string,
    public readonly date: string, // YYYY-MM-DD
  ) {}
}
```

**Handler behavior:** call `repo.findByUserAndDate(userId, date, tenantId)`, throw `NotFoundException` if null.

## Repository Implementation Notes

Use Drizzle's `generate_series` equivalent: since Drizzle doesn't natively support PostgreSQL `generate_series`, use raw SQL fragments via `sql` template tag:

```ts
import { sql } from 'drizzle-orm'

// Calendar spine — generate a row per day even without check-in
const calendar = db.execute(sql`
  SELECT generate_series(
    ${fromDate}::date,
    ${toDate}::date,
    '1 day'::interval
  )::date AS day
`)
```

Then left join `time.time_sheet` on `(day = date AND user_id = userId AND tenant_id = tenantId)`.

For the member and all-views, the enrichment data (name, badge_number, manager_name) comes from `PeopleQueryFacade` in the handler, not from a DB join. The repository returns raw `TimesheetCalendarRow[]` with `userId`; the handler enriches them with people data.

## tRPC Procedures

Add to `interface/trpc/time.router.ts`:

```ts
export const timeRouter = router({
  attendance: router({
    getMyTimesheet: protectedProcedure
      .input(
        z.object({
          fromDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
          toDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
          limit: z.number().int().min(1).max(100).default(20),
          offset: z.number().int().min(0).default(0),
        }),
      )
      .query(({ ctx, input }) =>
        svc().query(
          new GetMyTimesheetQuery(
            ctx.tenantId,
            ctx.actorId,
            input.fromDate,
            input.toDate,
            input.limit,
            input.offset,
          ),
        ),
      ),

    getMemberTimesheet: protectedProcedure
      .meta({ permission: 'time:attendance:team:read' })
      .input(
        z.object({
          fromDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
          toDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
          badgeNumber: z.string().optional(),
          limit: z.number().int().min(1).max(100).default(20),
          offset: z.number().int().min(0).default(0),
        }),
      )
      .query(({ ctx, input }) =>
        svc().query(
          new GetMemberTimesheetQuery(
            ctx.tenantId,
            ctx.actorId,
            input.fromDate,
            input.toDate,
            input.badgeNumber,
            input.limit,
            input.offset,
          ),
        ),
      ),

    getAllTimesheet: protectedProcedure
      .meta({ permission: 'time:attendance:all:read' })
      .input(
        z.object({
          fromDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
          toDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
          badgeNumber: z.string().optional(),
          name: z.string().optional(),
          limit: z.number().int().min(1).max(100).default(20),
          offset: z.number().int().min(0).default(0),
        }),
      )
      .query(({ ctx, input }) =>
        svc().query(
          new GetAllTimesheetQuery(
            ctx.tenantId,
            ctx.actorId,
            input.fromDate,
            input.toDate,
            input.badgeNumber,
            input.name,
            input.limit,
            input.offset,
          ),
        ),
      ),

    getEntry: protectedProcedure
      .input(
        z.object({
          userId: z.string().uuid(),
          date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
        }),
      )
      .query(({ ctx, input }) =>
        svc().query(new GetTimesheetEntryQuery(ctx.tenantId, input.userId, input.date)),
      ),
  }),
})
```

**Role permission mapping:**

- `time:attendance:own:read` — all authenticated users (staff, manager, hrm)
- `time:attendance:team:read` — manager, hrm
- `time:attendance:all:read` — manager, hrm
- The kernel permission system enforces these; handlers do not re-check roles
