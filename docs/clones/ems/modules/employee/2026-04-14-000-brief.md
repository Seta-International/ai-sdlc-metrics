# Clone Brief: employee

## Business Purpose

The People module is the core of the HRM platform — it manages the full employee lifecycle from pre-hire to termination and rehire. It serves as the system of record for who works at the company, what they do, where they sit in the org, and how their employment evolves over time.

Unlike the legacy EMS (basic CRUD with a fragile draft system), the target People module is designed as an HRM-grade system informed by BambooHR, HiBob, Personio, and Rippling patterns: effective-dated position history, per-field change approvals, JSONB profile sections, field-level access control, and LinkedIn-powered onboarding.

## Decisions

| Feature                                  | Decision      | Rationale                                                                                                                                                   |
| ---------------------------------------- | ------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Job history (effective-dated)            | **new**       | #1 missing feature across all HRMs. Promotions, title changes, dept moves are lost without this. Closed-open interval `[effective_date, end_date)` pattern. |
| Employment lifecycle state machine       | **reimagine** | Enum exists but no guarded transitions or event emission. Every transition must be recorded as an immutable event.                                          |
| Employee directory + search              | **new**       | Full-text search across name/email/title/department/skills. Filterable. Industry table stakes.                                                              |
| Profile change requests                  | **reimagine** | Target already has per-field model (better than legacy). Enhance with batch approval, effective dating, and edit policy per field.                          |
| Employee documents                       | **new**       | File references with category, expiry, confidentiality. Uses `@future/storage`. Standard in all HRMs.                                                       |
| Custom fields per tenant                 | **new**       | Definition table + JSONB. Every mid-market HRM has this. Avoids constant feature requests for tenant-specific data.                                         |
| Probation management                     | **new**       | Track period, pg-boss reminders to manager, record confirmation/extension/termination outcome.                                                              |
| LinkedIn profile import                  | **new**       | OAuth-based import of education, work experience, skills, certs. Massive onboarding UX improvement.                                                         |
| Share link                               | **keep**      | JWT-based profile sharing for external use (e.g., sending to clients). Simple, useful.                                                                      |
| Company email generation                 | **keep**      | Vietnamese name → latin transliteration → `first.last@domain`. Established business rule.                                                                   |
| Duplicate field validation               | **keep**      | Email, phone, identity numbers unique across tenant. Critical data integrity.                                                                               |
| Field-level access control               | **new**       | 3 tiers: public (all employees), restricted (HR + manager), confidential (HR only). Applied at query level.                                                 |
| Profile completeness                     | **new**       | Computed score, nudges for missing sections, configurable required sections. Drives data quality.                                                           |
| Bulk operations + CSV import             | **new**       | HR manages hundreds of people. Bulk update, CSV import with preview/validate/commit.                                                                        |
| Rehire support                           | **new**       | New `employment_profile` linked to same `actor_id` with `previousProfileId` backlink.                                                                       |
| Legacy draft workflow (10 shadow tables) | **skip**      | Target's per-field `profile_change_request` is already superior.                                                                                            |
| Legacy effort tracking                   | **skip**      | Belongs in `projects` module, consumed via `ProjectsQueryFacade`.                                                                                           |
| Legacy salary data                       | **skip**      | Belongs in `finance` module.                                                                                                                                |
| Legacy avatar sync (MS Teams)            | **skip**      | Identity module concern.                                                                                                                                    |
| Legacy employee ID suggestions           | **skip**      | UUID v7 replaces numeric IDs. `employeeCode` stores human-readable code.                                                                                    |
| Legacy delete-recreate update pattern    | **skip**      | Anti-pattern. Use in-place updates + soft delete.                                                                                                           |

## Risks & Assumptions

- **Job history backfill:** Existing profiles have no position history. Migration must seed one `job_history` row per profile with current values and `effective_date = hireDate`.
- **LinkedIn API limits:** LinkedIn's API is restrictive (requires partner program for profile data access). May need to start with manual URL entry + optional import, or use a third-party enrichment service.
- **Custom fields performance:** JSONB queries with GIN index are fast for equality but slow for range queries on custom date fields. Acceptable for mid-market scale.
- **Profile section schema validation:** JSONB payloads need Zod schemas per section type to prevent garbage data. Each new section type requires a corresponding schema.
- **Effective dating complexity:** Closed-open intervals need careful query logic. Use a shared utility for "as-of" queries.
- **Field access control granularity:** Starting with 3 tiers (public/restricted/confidential) at the field category level, not per-individual-field. Per-field config is post-MVP.
