# Clone Brief: project-staffing

## Business Purpose

The Projects module manages client engagements, project teams, and resource planning for consulting/IT services firms. It answers the three questions resource managers ask daily: "What do our projects need?", "Who's available?", and "Are we utilizing our people well?"

Unlike the legacy EMS (basic project CRUD + flat employee allocation), the target is a PSA-grade (Professional Services Automation) system informed by Kantata, Productive.io, Certinia, Float, and Runn: demand-slot-based staffing, project org charts separate from HR hierarchy, skills-based resource matching, and capacity analytics.

**Scope boundary:** This module owns project setup, team structure, and resource planning. Financial tracking (rate cards, budgets, profitability) belongs in `finance`. Timesheets belong in `time`. Delivery management (phases, milestones, gates, risks) is deferred — Jira handles delivery for now.

## Decisions

| Feature                                 | Decision               | Rationale                                                                                                                                                                                                             |
| --------------------------------------- | ---------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Project Charter                         | **new**                | Lightweight project definition (objectives, scope, stakeholders). Every PSA captures structured project setup beyond name/dates. 1:1 detail table keeps project listing lean.                                         |
| Project Members (org chart)             | **new**                | Project reporting lines separate from HR hierarchy. PM → Tech Lead → Developers. Critical for consulting firms where project teams have their own structure. Kantata/Workday both model this as separate from HR org. |
| Team Templates                          | **new**                | Reusable team composition templates ("Scrum Team" = 1 PM + 1 BA + 4 devs + 1 QA). Auto-populate ProjectRole demand slots on project creation. Speeds setup, enforces standards.                                       |
| Skills-based resource matching          | **new**                | Match ProjectRole.skillsRequired against employee skills from `people` module. Rank candidates by fit + availability. Core PSA differentiator.                                                                        |
| Staffing analytics completion           | **keep**               | Queries exist but return empty arrays. Wire PeopleQueryFacade to complete staffing overview and capacity report.                                                                                                      |
| Bench & utilization queries             | **new**                | "Who's idle?" and "What's our billable utilization?" are the #1 reports for resource managers. Every PSA has them.                                                                                                    |
| Account/project lifecycle               | **keep**               | Missing delete/archive commands, project name uniqueness, self-removal guard. Basic CRUD completeness.                                                                                                                |
| Enhanced allocation workflow            | **reimagine**          | Add `proposed` status (doesn't count against capacity, used for planning). Allocation conflict detection (warn when over standard hours).                                                                             |
| Batch operations                        | **new**                | Batch allocation creation, batch close. HR/PM needs to staff multiple roles at once.                                                                                                                                  |
| Cross-module facades & events           | **new**                | Expose ProjectsQueryFacade methods for time/finance/planner. Emit new events for cross-module reactions.                                                                                                              |
| Phases, milestones, gates               | **skip (later phase)** | Jira handles delivery management now. Will add when building native delivery tracking.                                                                                                                                |
| Deliverables & review workflow          | **skip (later phase)** | Same — delivery scope deferred.                                                                                                                                                                                       |
| Status reports & RAG health             | **skip (later phase)** | Same — delivery scope deferred.                                                                                                                                                                                       |
| Risk & issue register                   | **skip (later phase)** | Same — delivery scope deferred.                                                                                                                                                                                       |
| Legacy developer settings (JIRA/GitHub) | **skip**               | Credentials in DB is anti-pattern. Out of scope.                                                                                                                                                                      |
| Legacy effort % model                   | **skip**               | Target's hoursPerDay is better — handles part-time and variable schedules.                                                                                                                                            |
| Legacy permission-aware listing         | **skip**               | Target delegates to kernel Authority system. Correct architecture.                                                                                                                                                    |

## Risks & Assumptions

- **PeopleQueryFacade dependency:** Staffing analytics and resource matching depend on `PeopleQueryFacade.listActiveActors()` and skills queries. People module must expose these facades first (or implement concurrently).
- **Standard hours assumption:** Capacity calculations assume 8h/day standard. Should be configurable per tenant or per person (from `time` module work schedule).
- **Skills taxonomy:** Resource matching quality depends on consistent skills data in `people` module. No shared skills taxonomy exists yet — matching will use string comparison initially.
- **Project member vs allocation:** Two separate concepts (org chart vs capacity). Users may confuse them. Clear UI/API naming is critical.
