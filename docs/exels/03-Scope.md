## Decomposition convention

Tracks split **horizontal vs vertical**:

- **Horizontal tracks — Core Backend, Core Frontend, Core AI Agent.** Monorepo scaffolding, shared auth, common components/utilities, routing, caching, base observability, permission primitives, LLM plumbing. **No module-specific code.** Built once; reused by every Future module.
- **Vertical track — Planner.** Integrates the three Core horizontal slices into a user-facing product. **Every Planner feature has its own BE handlers, FE UI, and (where relevant) Agent-integration sub-work.**

The Planner WBS therefore contains Planner-BE rows, Planner-FE rows, and Planner-Agent-integration rows. The Core tracks contain only horizontal primitives.

## Block 1 — WBS-Master (feature-level rollup)

Effort in MD (man-days). Confidence: **H** = well-understood, **M** = some novelty, **L** = material unknowns. Layer: **BE** / **FE** / **AI** / **BE+FE** / **BE+AI** / **FE+AI** indicates which technical layer(s) the feature spans. Sprint column targets S1–S4 (W1–W8).

**Ownership convention:** `Owner = FS#1` or `Owner = FS#2` identifies a single full-stack dev who owns that feature **end-to-end — BE + FE + any integration layer.** There is no BE-vs-FE split between the two FS devs. Per-feature ownership is provisional in the Owner column and finalised at Kickoff. `Owner = AI Eng` identifies the AI Engineer as end-to-end owner (including the BE hooks needed to integrate). Non-builder rows (PM / PO / SM / BA / QA) carry the respective role name.

## Block 4 — Change Request (CR) Process

1. **Submit CR.** Requester completes CR form; assigned CR ID.
2. **Impact Analysis.** Track lead + PM assess scope / time / cost / risk.
3. **Approval.** CCB (Sponsor + PMO + CTO) or PM (under veto authority) approves.
4. **Re-baseline.** WBS, timeline, acceptance weights updated on this workbook.
5. **Execute.** Work performed; CR row status updated until closed.

**Rule:** No CR = No work. Every change must be traceable (ID, owner, date, decision).
