# SCOPE — platform/tenant  (@seta/tenant)

## Purpose

`@seta/tenant` owns request-identity propagation across async boundaries inside
the API process. It exposes an AsyncLocalStorage-backed `tenantContext`, a Hono
middleware that establishes that store from a resolver, and the Drizzle schema
for `tenant.tenants` + `tenant.tenant_connectors`. It is the **only** sanctioned
way to read the current `tenantId` / `userId` anywhere else in the codebase —
every other package (`@seta/db`, `@seta/oauth`, `@seta/middleware` logger, agent
runtime, connectors) reads from this ALS rather than accepting tenant ids as
function parameters.

## Responsibilities

- **Owns:**
  - The `AsyncLocalStorage<TenantContextStore>` instance and its public API
    (`tenantContext.run` / `getTenantId` / `getUserId`).
  - The Hono middleware shape that bridges an HTTP request into the ALS store
    (resolver injected by the caller — `@seta/tenant` does not know about JWTs,
    headers, or API keys).
  - The Drizzle schema for the `tenant` Postgres schema (`tenants`,
    `tenant_connectors`) plus their inferred row/insert types.
- **Does NOT own:**
  - The `withTenant` DB wrapper / `SET LOCAL app.tenant_id` plumbing — that
    lives in `@seta/db` (see setup.md §3 "The only correct way to set the GUC").
  - Resolving identity from a request (JWT verification, API-key lookup,
    Bot-Framework SSO assertions). Callers supply a `resolve(c) => …` function
    to `tenantMiddleware`.
  - RBAC, scopes, role checks. Those belong in `@seta/auth` and route
    middlewares.
  - User/session storage. `auth.users` and `auth.sessions` are owned by
    `@seta/auth`; this package only models the *tenant* aggregate.

## Current state (Epic 1)

Implemented and shipped on `main`:

- `src/context.ts` — `tenantContext` object with `run<T>`, `getTenantId()`
  (throws `Unauthorized` from `@seta/middleware` if no active store — matches
  the deny-by-default contract called out in spike report 07), and
  `getUserId()`.
- `src/middleware.ts` — `tenantMiddleware(resolve)` factory; returns 401
  `application/problem+json` when the resolver yields `null`, otherwise enters
  `tenantContext.run` for the rest of the request chain.
- `src/schema.ts` — `tenantSchema` (`pgSchema('tenant')`) with `tenants` and
  `tenant_connectors` tables; types via `$inferSelect` / `$inferInsert`.
- `src/index.ts` — re-exports the three above plus `TenantContextStore` type.
- `src/context.test.ts` — unit tests covering deny-by-default, single-`run()`,
  and nested-`run()` override semantics.
- `migrations/0000_amazing_lady_ursula.sql` — generated by drizzle-kit; creates
  schema, both tables, and the FK from `tenant_connectors.tenant_id`.

## Public interface

- `type TenantContextStore = { tenantId: string; userId?: string }` — the ALS
  payload shape. **Frozen-in-spirit**: never mutate after `run()`.
- `tenantContext.run<T>(store: TenantContextStore, fn: () => Promise<T>): Promise<T>`
  — only sanctioned entry point that pushes a store onto the ALS frame.
- `tenantContext.getTenantId(): string` — throws `Unauthorized('no tenant
  context')` if called outside an active `run()`.
- `tenantContext.getUserId(): string | undefined` — non-throwing.
- `tenantMiddleware(resolve: (c) => Promise<{tenantId; userId?} | null>): MiddlewareHandler`
  — Hono middleware factory.
- `tenants`, `tenantConnectors`, `tenantSchema` — Drizzle table builders for
  the `tenant` Postgres schema.
- `Tenant`, `NewTenant`, `TenantConnector`, `NewTenantConnector` — inferred
  row / insert types.

## Imports

- **Allowed internal:**
  - `@seta/middleware` — for `Unauthorized` (used by `getTenantId` deny path).
- **Forbidden:**
  - `@seta/db`, `@seta/auth`, `@seta/oauth`, `@seta/audit`,
    `@seta/connector-registry`, anything under `modules/*` or `apps/*`. This
    package is upstream of every other tenant-aware package; importing them
    would create a cycle. `@seta/db.withTenant` will read from this ALS but
    `@seta/tenant` must not call back into `@seta/db`.
- **External (pinned per setup.md §13):**
  - `hono@4.12.18` — for `MiddlewareHandler` type only.
  - `drizzle-orm@0.45.2` — for `pgSchema` / column builders.
  - `dotenv@17.4.2`, `zod@4.4.3` — present in `package.json` but currently
    unused by `src/*`; retained for migration runner consistency.

  **Divergence from setup.md §13:** §13 line 1822 lists `@seta/tenant` with no
  external runtime deps beyond `node:async_hooks`. Reality adds `hono`,
  `drizzle-orm`, `zod`, `dotenv` because middleware + schema were folded into
  the same package rather than split.

## Patterns to follow

- **Deny-by-default `getTenantId()` that throws, never returns `undefined`** —
  spike report 07 §"Punch list" bullet 1
  (`docs/explorations/2026-05-12-mastra-spike/07-request-context.md:36`) and
  setup.md §3 "API rule: Tenant id | Never a function param"
  (`docs/setup.md:2063`). Current `src/context.ts:20-22` does this.
- **Store is logically frozen for the lifetime of a request** — spike report
  07 §Delta "Mastra's mutable `set()` … should be frozen after creation in
  the auth middleware" (`07-request-context.md:26`). Never expose a `set()` /
  `update()` / `promoteUser()` helper. New fields land via a new
  `tenantContext.run` frame, not by mutating an in-flight store.
- **Resolver injection in the middleware** keeps `@seta/tenant` free of any
  knowledge about JWT/Bot-Framework/API-key validation — the consumer in
  `apps/api/src/main.ts` (or a sibling auth middleware) owns identity proof.
  See `src/middleware.ts:9-23` and CLAUDE.md "No DI containers" rule.
- **Drizzle schema-per-module** — own `drizzle.config.ts` + `migrations/`;
  cross-context references go by tenant id only, never via cross-schema FKs.
  Setup.md §3 "Schema-per-module (DDD)" (`docs/setup.md:102`).
- **Nested `run()` overrides parent** — verified by `src/context.test.ts:17-24`
  and matches AsyncLocalStorage semantics; intentional for queue handlers /
  background jobs that legitimately need to re-enter as a different tenant.

## Patterns to avoid

- **Do not use the ALS store as a DI bag.** Spike report 07 §Delta "Avoid …
  DI/RequestContext conflation. Mastra uses RequestContext as a DI bag … Don't"
  (`07-request-context.md:27`). Services come from explicit imports, not from
  `tenantContext`.
- **Do not pass `tenantId` as a function parameter** to anything below the
  middleware. Setup.md §15 "Tenant id | Never a function param"
  (`docs/setup.md:2063`). The one exception is the OAuth admin-consent
  callback path where the tenant id arrives in the query string *before*
  `tenantContext.run` has been entered — see `@seta/oauth.completeAdminConsent`.
- **Do not call `getTenantId()` outside an active `run()`** expecting a
  fallback. Failures here are intentional (RLS denial is the desired surface).
  Spike report 07 §"What setup.md plans" quoting setup.md §3 "Anything that
  does `await sql\`SELECT ...\`` directly … will have `app.tenant_id` unset —
  RLS will reject the query. That's the desired failure mode: deny by default."
- **Do not import this package from a browser-bound module.** It uses
  `node:async_hooks` and will break Studio (P2). Spike report 07 punch-list
  P2-defer bullet (`07-request-context.md:46`).
- **Do not add a cross-schema FK from `tenant_connectors` to any other
  module's schema** — CLAUDE.md "No cross-schema foreign keys; cross-context
  references by ID only."

## Test strategy

- **Unit (current):** `src/context.test.ts` covers the three load-bearing
  invariants — throw-when-empty, single-run lookup, nested-run override.
  Vitest 4.1.5, no Postgres required.
- **Middleware unit (gap):** no test covers `tenantMiddleware` 401 path or
  resolver-throws path. Worth adding when a second middleware caller lands.
- **Integration (deferred):** spike report 07 punch-list bullet 4
  (`07-request-context.md:39`) asks for an integration test that
  `getTenantId()` survives across `postgres-js sql.begin` and
  `msal-node.acquireTokenOnBehalfOf` — currently exercised indirectly through
  `@seta/oauth/src/refresh.test.ts` (which depends on `DATABASE_URL`) but not
  asserted as a direct contract. Open.

## Open questions

1. Should `tenantContext.run` deep-freeze the store object so callers can't
   reach in and mutate `userId`? Spike report 07 §Delta argues yes; current
   impl relies on convention.
2. Background-job entrypoints (Epic 3 sync workers, cron) currently call
   `tenantContext.run` directly. Spike report 07 punch-list bullet 3
   (`07-request-context.md:38`) proposes a sibling `runAsTenant(tenantId, fn)`
   helper that audit-logs the entry — defer until the first non-HTTP caller
   appears.
3. SSE streams (`streamKernelSSE` in `@seta/agent-core`) — does the ALS frame
   survive into each chunk producer's `pull()` callback, or does the kernel
   need to re-enter per chunk? Spike report 07 Open Question 2
   (`07-request-context.md:31`). Needs a one-off probe before K3 ships.
4. `tenants.status` and `tenant_connectors.status` are free-form `text`
   today. Worth narrowing to a Zod-validated enum at the service boundary
   once a second consumer (sync worker) reads them.
