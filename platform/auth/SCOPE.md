# SCOPE — platform/auth  (@seta/auth)

## Purpose

`@seta/auth` owns the canonical user/session/api-key identity store for Seta —
the Drizzle schema for the `auth` Postgres schema and (forthcoming, per
setup.md §4) the argon2-based API-key hashing/verify path with
upgrade-on-verify, plus a `KmsProvider` interface that abstracts AWS KMS /
Azure Key Vault / `EnvDekProvider` for at-rest encryption of secrets. It is
the only place that owns `auth.users`, `auth.sessions`, `auth.api_keys`.

## Responsibilities

- **Owns:**
  - `auth.users` (canonical Seta user identity — one row per person per
    tenant, with `external_provider` / `external_subject` for SSO mapping).
  - `auth.sessions` (Postgres-backed server sessions for the P2 inbound SSO
    flow).
  - `auth.api_keys` (hashed API keys, per-tenant scopes, revocation
    timestamp).
  - The argon2id hash/verify helper with `needsRehash` upgrade-on-verify
    (forthcoming; spec at setup.md §4 lines 245-275).
  - The `KmsProvider` interface and its AWS / EnvDek implementations
    (forthcoming; spec at setup.md §4 lines 277-325). **Note:** AWS KMS +
    EnvDek shipped in Epic 1 but currently live in `@seta/oauth/src/kms.ts`;
    the long-term home per setup.md is `@seta/auth/src/kms/`. See Open
    Questions.
- **Does NOT own:**
  - Tenant identity / `tenant.tenants` — that's `@seta/tenancy`.
  - OAuth token storage, MSAL CCAs, admin-consent URL building — that's
    `@seta/oauth`.
  - Authorization (RBAC). RBAC will land here per setup.md §13 ("RBAC +
    KmsProvider"), but the current `users.status` text column is not a
    role/scope axis.
  - JWT verification. `jose`-based Bot Framework JWT verify lives in
    `modules/channels/teams` and any P2 SSO ID-token validation lives in a
    sibling sso package. `@seta/auth` does not depend on `jose`.
  - Session middleware / cookie signing. The `auth.sessions` row schema is
    here; the route handlers that mint/validate them are not.

## Current state (Epic 1)

Implemented and shipped on `main`:

- `src/schema.ts` — Drizzle schema with three tables:
  - `users` (`id`, `tenantId`, `externalProvider`, `externalSubject`, `email`,
    `displayName`, `status`, timestamps) + two unique indexes:
    `(tenantId, email)` and `(externalProvider, externalSubject)`.
  - `sessions` (`id`, `tenantId`, `userId`, `expiresAt`, `createdAt`).
  - `apiKeys` (`id`, `tenantId`, `hashedKey`, `scopes text[]`, `createdAt`,
    `revokedAt`).
  - Inferred types `User` / `NewUser` exported.
- `src/index.ts` — re-exports everything from `./schema`.
- `migrations/0000_brown_blonde_phantom.sql` — generated by drizzle-kit.
- `package.json` declares `@node-rs/argon2@2.0.2` and `lru-cache@11.3.6` as
  runtime deps **but no source consumes them yet** — the hash/verify helper
  and any in-memory key-verify LRU are not yet implemented.
- No `KmsProvider` interface lives here today; `@seta/oauth/src/kms.ts`
  ships the AWS + EnvDek implementations. See Open Questions.

## Public interface

- `auth` — `pgSchema('auth')` builder.
- `users`, `sessions`, `apiKeys` — Drizzle table objects (read by services
  in `apps/api` and the SSO product).
- `User`, `NewUser` — `$inferSelect` / `$inferInsert` types for `users`.
- **Not yet exported (planned per setup.md §4):**
  - `hashApiKey(raw: string): Promise<string>`
  - `verifyApiKey(raw, stored, onUpgrade): Promise<boolean>` with
    `needsRehash` auto-upgrade.
  - `KmsProvider` interface + `AwsKmsProvider` / `AzureKeyVaultProvider` /
    `EnvDekProvider` impls.
  - `EncryptionContext` type bound to `{ tenantId, purpose }` for AAD.

## Imports

- **Allowed internal:**
  - `@seta/db` — for `tenantUser` role export when RLS policies are added to
    `auth.api_keys` / `auth.sessions` (setup.md §3 expects RLS on every
    tenant-data table; not yet wired).
- **Forbidden:**
  - `@seta/tenancy` — circular risk; `@seta/auth` does not read the ALS, it
    only models persistence. Services in `apps/api` join the two.
  - `@seta/oauth` — `auth` is upstream of `oauth` in the dep DAG (oauth
    depends on a user existing).
  - `@seta/middleware` — currently no `Unauthorized` throws in this package;
    if/when API-key verify lands, throw `DomainError` subclasses from
    `@seta/middleware/errors` per CLAUDE.md, but verify can also return
    `false` and let the route translate.
  - Anything under `modules/*` or `apps/*`.
- **External (pinned per setup.md §13):**
  - `@node-rs/argon2@2.0.2` — declared, not yet imported.
  - `lru-cache@11.3.6` — declared, intended for an API-key verify cache
    keyed on `hashedKey` to skip argon2 re-runs on repeated requests.
  - `drizzle-orm@0.45.2`, `zod@4.4.3`, `dotenv@17.4.2`.

  **Divergence from setup.md §13:** setup.md §13 lines 1822-1832 list
  `@seta/auth` as owning argon2 hashing + `KmsProvider` + AES-GCM. Reality
  in Epic 1: only the schema landed here; argon2 helpers are declared as
  deps but unused, and KMS lives in `@seta/oauth/src/kms.ts`. The expected
  end state matches setup.md §4 — see Open Question 1.

## Patterns to follow

- **`@node-rs/argon2`, never `argon2`** — setup.md §4 row 180 explicitly:
  "Native (Rust binding via napi-rs, NOT the C-based `argon2` package)." The
  C package needs node-gyp and breaks in slim Docker images.
- **OWASP 2024 PARAMS pinned at the top of the helper module** — setup.md §4
  lines 253-259 specifies `argon2id, m=64MB, t=3, p=4, hashLength=32`. When
  these tighten, only this constant changes; `needsRehash` then upgrades
  stored hashes on next verify.
- **`needsRehash` + upgrade-on-verify** — setup.md §4 lines 263-272. The
  verify callsite supplies an `onUpgrade(newHash)` callback so persistence
  is the caller's responsibility, keeping `@seta/auth` independent of any
  particular `users`-row update path.
- **`EncryptionContext` is part of the `KmsProvider` interface** —
  setup.md §4 line 291: `{ tenantId, purpose }` becomes AAD on KMS
  `EncryptionContext`. Defeats bulk-dump attacks even when a KMS Decrypt
  permission leaks.
- **`KmsProvider` is selected by `KMS_PROVIDER` env** — setup.md §4 line
  279. `EnvDekProvider` for local dev, `AwsKmsProvider` / Azure for cloud.
- **`Zod schema → TS types via z.infer`, never a parallel `interface`** —
  CLAUDE.md "Schema-driven — always generate." The `users.status` etc.
  text columns should narrow to a Zod enum at the service boundary, not by
  introducing a parallel TS type here.
- **Drizzle schema → migration SQL via `drizzle-kit generate`** — CLAUDE.md
  "Never hand-edit migrations/*.sql; fix the schema and regenerate."
- **Schema-per-module** — own `drizzle.config.ts` + `migrations/`; no
  cross-schema FKs; `users.tenant_id` is by id, not a Drizzle `.references`
  to `@seta/tenancy.tenants` (CLAUDE.md "No cross-schema foreign keys").

## Patterns to avoid

- **Do not ship a JWT verifier here.** `jose` is the project pick
  (setup.md §4 row 185) but it lives in the Teams adapter (Bot Framework
  inbound) and any future SSO product — not in `@seta/auth`. `@seta/auth`
  has no `jose` dep and should not gain one.
- **Do not duplicate the KMS impl.** If `KmsProvider` migrates here from
  `@seta/oauth`, delete the oauth copy in the same PR (CLAUDE.md "No
  legacy, no backward compat. Change every caller in the same PR and
  delete the old shape").
- **Do not import `@seta/tenancy`** to read `tenantId` — `auth` is a
  persistence-only module. `apps/api` services pass `tenantId` to `auth`
  helpers; that is *not* the "never a function parameter" rule (setup.md
  §15 line 2063) because the call site itself reads the ALS via
  `tenantContext.getTenantId()`.
- **Do not bypass `withTenant` when querying `auth.users` etc.** Once RLS
  policies land on `auth.*` (CLAUDE.md "Every tenant-data table has an RLS
  policy"), unwrapped queries will fail by design. The pattern is set by
  `@seta/oauth/src/vault.ts:56-67`.
- **Do not roll your own password/secret hash.** Use the `@node-rs/argon2`
  primitives. Setup.md §4 explicitly pins this; spike report 02 §Delta
  Patterns to follow mirrors the "thin wrapper, no inline crypto"
  philosophy.
- **Do not bake a global module-scope LRU into the verify helper unless it
  is tenant-scoped** — CLAUDE.md "Shared cross-instance state must be
  Redis-ready shape today (typed key, TTL, tenant-scoped)."

## Test strategy

- **Unit (gap):** no tests ship in Epic 1 (`scripts.test:unit` is absent
  from `package.json`). When argon2 helpers land they need:
  - Known-vector round-trip (hash → verify true; hash → verify with wrong
    raw false).
  - `needsRehash` returns false when PARAMS unchanged; true when memoryCost
    changes; `onUpgrade` is invoked exactly once on success.
  - Constant-time-ish behaviour assertion (verify wrong raw still consumes
    ≥1 argon2 hash so we don't leak existence by timing).
- **Integration:** once RLS lands on `auth.api_keys`, an integration test
  using `tenant_user` role should assert cross-tenant SELECT returns zero
  rows — mirrors the pattern in `@seta/oauth/src/vault.test.ts`.
- **No live KMS calls in CI** — `EnvDekProvider` is the test-only impl
  (setup.md §4 line 323).
- **Vitest 4.1.5** with leaf override only on `test.name` (CLAUDE.md
  "Vitest config: root owns pool / coverage / thresholds / projects").

## Open questions

1. **KMS location.** Setup.md §4 lines 277-325 places `KmsProvider` in
   `@seta/auth/src/kms/`. Epic 1 shipped it in `@seta/oauth/src/kms.ts`
   because the first consumer was the OAuth token vault. Two options:
   (a) move it to `@seta/auth` and have `@seta/oauth` depend on
   `@seta/auth`; (b) leave it in `@seta/oauth` and revise setup.md §4.
   No deprecation aliases either way (CLAUDE.md "no compat shims").
2. **RBAC primitive shape.** `package.json:description` says "RBAC +
   KmsProvider" but no role/scope model lives here yet. Likely shape:
   `scopes: string[]` on `api_keys` plus a `requireScope(scope)`
   middleware factory in `@seta/middleware`, but the decision belongs to
   the SSO milestone (P1 plan dual-language architecture commit
   `b2c744cb`).
3. **`auth.users.external_provider` / `external_subject` enum.** Currently
   free-form text. The first external IdP to land (Entra via Bot
   Framework OBO) needs to pick a canonical provider id; coordinate with
   `@seta/oauth` and `modules/channels/teams`.
4. **Cross-schema FK ban vs `auth.users.tenant_id`** — `users.tenant_id`
   has no Drizzle `.references` to `tenant.tenants` (correct per CLAUDE.md
   "no cross-schema FKs"). Add a row-existence check or a deferred service
   guard in `apps/api`.
